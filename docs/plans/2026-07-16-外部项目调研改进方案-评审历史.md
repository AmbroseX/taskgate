# taskgate 外部项目调研改进方案——评审历史与否决方案存档

主文档:`2026-07-16-外部项目调研改进方案.md`。本文件存放逐轮修订说明和被否决的 v1 设计草案(主文引用的"附录 A.x"即本文件章节),主文只留当前结论、证据和待裁决项。

## v4 修订说明(2026-07-16,第三轮评审)

主问题 7 条 + 次要 5 条,全部成立,逐条落地:

1. **固定窗口在多机下不一定全局一致**:对齐 epoch 只保证公式相同,不保证时刻相同——多机时钟偏差下窗口边界附近两个 key 同时放量。5.3 新增"窗口时间用谁的钟"裁决(硬配额须服务端时间,与 fakeclock 铁律的调和是 research 必答;尽力配额可用节点钟但偏差要并进超发合同);5.6 的 fakeclock 表述跟着改;裁决表新增 #5。
2. **"超发上界 ≈ 进程数"不是可靠合同**:真实边界是"同时处于检查-扣减窗口内的认领循环实例数"(一个进程可建多个 Gate,重连/超时扩大不确定性,时钟偏差叠加)。5.4.2/5.5/裁决表 #2 全部改为"须先给并发模型与形式化上界,否则不能写成公开保证"。
3. **"RPS 退化会经 429 自纠正"论据删除**:`ErrThrottled` 只退避当前任务,不降低本地 limiter 速率,队列有存量时各进程仍按各自 RPS 发。5.4.3(b) 对比表改为"有没有立刻可见的信号"。
4. **"多进程 sqlite 配配额则 New() fail-fast"撤回**:进程无法感知别的进程打开同一库文件,判定不了。现实三选:sqlite 一律拒绝 / 显式 deployment scope 配置 / sqlite 也做库内共享计数。倾向改为"各后端在自己介质内做共享计数"。
5. **Replay 避免了重绑定但没保住历史**:Task 只有一份可变 Result/LastError/计数(taskgate.go:67),重跑后旧子任务按父 ID 查到的是新结果。6.4 把 e 收窄为 Retry-failed-only,并补入候选 f:**稳定业务键 + 不可复用 execution ID + `replay_of`**(第三轮评审指出的最常见模型),倾向改为"f 为长期基准模型,e 为短期解"。
6. **WaitToken 白烧的引用写错**:实际顺序 AcquireSlot → WaitToken → Dequeue(scheduler.go:248),白烧发生在拿到令牌后 Dequeue 扑空/出错(scheduler.go:262);"先拿令牌再等槽"是注释解释为什么不那么做的反例。5.5 修正。
7. **7.5.2 表格通知分类回归**:又把"发通知"放回本地事务可防一列。已同步 README 的三分类(本地数据库 / 外部网关 / 异步消息通知)。

次要:①"配额行锁划算"降级为初步判断,认领热路径全消费者争同一行,须压测定论;②OnStateChange"多进程各跑一次"改为"回调只在执行流转的进程本地触发,真正问题是可能丢失、随重跑重复、与落库不同事务";③修订历史与附录 A 迁入本文件,主文瘦身;④8.3/10 节两处"三后端"改"五后端";⑤10 节澄清"不做 FaaS"= 库不变成 FaaS 平台,独立平台内嵌 taskgate(Serverless 方案的 Trigger Service)不冲突。

**战略调整(采纳评审结论)**:下一轮 research 不再横向比接口,先收敛两个领域模型——Quota{窗口时间来源、扣减单位、硬/软保证、故障语义}、Identity{业务幂等键、任务身份、执行尝试、Replay 历史},**Identity 优先**。已写进主文 12/13 节。

## v2 修订说明(2026-07-16,第一轮评审)

`review.md` 提了 13 条,**经逐条核实,13 条全部成立,全是本方案 v1 的错误**。其中三条是源码层面的判断错误:

| 评审项 | v1 的错误 | 核实证据 |
|---|---|---|
| **#8** | 把 `List` 的查询排序 `(CreatedAt, ID)` 当成了执行顺序,据此编造了"共享队列会先跑完所有检索"的结论 | Dequeue 按 `(RunAt, ID)`:`memorybroker/broker.go:232` 注释「取 RunAt 最早的(同刻取 ID 最小)」;`sqlitebroker/schema.sql` 的 `idx_claim ON tasks(queue, status, run_at)`。**整条撤回** |
| **#1** | 建议用 `OnStateChange` 承载扣费 | `broker.go:14` 合同为"可 nil、必须异步调";memory/sqlite/redis 三份实现(`memorybroker/broker.go:109`、`sqlitebroker/broker.go:160`、`redisbroker/broker.go:204`)均 `defer func() { _ = recover() }()` 静默吞 panic。**用它扣费比不防护更危险** |
| **#10** | 写租约回收延迟 ≤ `LeaseTTL`,漏算 reaper 扫描周期 | `scheduler.go:185` 是 `reapLoop(ctx, minTTL/2)` → 实际 `LeaseTTL` ~ `1.5×LeaseTTL` |

另外核实中发现**两处评审也没提到的问题**,一并修订:

- **`Delete` 的悬空引用有三处不是两处**:除 `task_deps` 的两个方向外,`tasks.depends_on` 还以 JSON 数组文本又存了一份(`sqlitebroker/enqueue.go:89`)
- **配额"退回进程内"的后果比评审说的更严重**:评审只提到 sqlite(同机多进程),但 `pgbroker`/`mysqlbroker` 已落地且定位是多进程/多机(`pgbroker/broker.go:6` 明确不实现 `LimiterProvider`)——进程数无上界,不是 4 倍是 N 倍

**v1 最该反省的一点**:第 7 节在纠正"恰好执行一次"这个措辞错误,而 v1 自己在 4.3 节写了"不用锁就达成恰好一次"(评审 #9)、在 README 示例里暗示本地幂等消除了重复调用(评审 #2)。**同一份文档里一边纠错一边复发同样的错**——"exactly-once"这类说法的传染性比想象的强,不是改几处措辞就完事的。

## 附录 A:v1 被否决方案存档

> 只作历史记录,**不是现行建议**。否决理由在主文 5.4.1/5.4.3 与 6.4。

### A.1 周期配额的 v1 设计(已否决)

**配置字段草案**(`QueueConfig` 加两个字段;裁决后要重新起草):

```go
// QuotaLimit 每个周期内最多放行多少个任务;0 = 不限配额。
QuotaLimit int `yaml:"quota_limit"`
// QuotaPeriod 配额周期。QuotaLimit>0 时必填。
QuotaPeriod Duration `yaml:"quota_period"`
```

校验规则草案:`QuotaLimit < 0` 报错;`QuotaLimit > 0 && QuotaPeriod <= 0` 报错;`QuotaLimit == 0` 忽略 `QuotaPeriod`。

**接口草案(否决,理由见主文 5.4.1:`CommitQuota` 无 ctx 无返回值,无法表达提交失败)**:

```go
WaitQuota(ctx context.Context) error
CommitQuota()
```

**实现草案**:本地 `localLimiter` 用进程内计数器 + 窗口起点时间戳;分布式由 `redisbroker` 实现 `LimiterProvider`,`INCR` + `EXPIRE` 到窗口末尾,键形如 `tg:quota:{queue}:{windowStart}`(否决点之二:INCR/EXPIRE 分两步有"队列永久锁死"的崩溃窗口,必须单条 Lua);scheduler 沿用 `broker.(LimiterProvider)` 能力断言。**"SQL 后端不实现、自动退回进程内配额"已被主文 5.4.3 否决**(多进程下配额 N 倍超发且无反馈信号)。此外 v3 发现**配额根本不能直接复用 `LimiterProvider`**(整包替换语义,见主文 5.4),v4 又撤回了配套的"多进程 sqlite fail-fast"(判定不了)。

### A.2 `Delete` 的 v1 设计(已否决)

**API 草案**:

```go
// Delete 删除一个终态任务,释放其 ID。
// 非终态任务返回 ErrNotFinal(想删先 Cancel);
// 仍有未终态子任务依赖它的返回 ErrHasDependents;
// 任务不存在返回 ErrTaskNotFound。
func (g *Gate) Delete(ctx context.Context, id string) error
```

约束草案:只能删终态(否则 `ErrNotFinal`,想删先 `Cancel`);被"还可能来读"的子任务依赖时返回 `ErrHasDependents`;删除同步扣 `Counts`。

**拦截判据草案(即主文说的"v1 判据",已否决)**:该任务存在至少一个**未到终态**的子任务 → 拒绝删除;子任务全部终态(或没有子任务)→ 允许删除。依据是 dask `local.py:290-317` `finish_task` 的引用计数对偶结构——`waiting` 空则可运行(对应 taskgate 已有的 `PendingParents`),`waiting_data` 空则可释放(taskgate 无对应物);判据用四份实现已有的 `children` 反向索引逐个查状态即可算出。**否决原因(主文 6.4)**:dask 的 `waiting_data` 是一次 `compute()` 内的瞬时状态,计算结束整个丢弃,没有"历史"和"key 复用"问题;taskgate 的任务是持久化历史且删除后 ID 可复用——按此判据删除后复用 ID,历史依赖 `C.DependsOn` 会静默指向新任务。

dask 分析中仍然成立、被主文 6.4 保留引用的一条:`and dep not in results`(用户显式要的结果不自动释放)映射到 taskgate = **不做自动回收,释放必须是用户的显式决定,不能是 Submit 的副作用**。
