## 1. 要解决的问题(一句话回顾)

现有 `Task.ID` 一个字段背了两种职责:**业务幂等键**(cron 配方的 `daily-report:2026-07-16`,用来去重)和**执行历史身份**(依赖引用、结果溯源)。两种职责的生命周期要求相反——幂等键需要"失败后可以再来",历史身份需要"永不改变"——于是产生了"终态 ID 永久占死"和"删除/复用会篡改历史"这对无解的矛盾。本模型把它们拆开。

## 2. 四个概念

| 概念 | 定义 | 生命周期 | 谁生成 |
|---|---|---|---|
| **BusinessKey** | 业务幂等键。用户语义的"这件事"(如 `daily-report:2026-07-16`) | 可重复提交:同一个键下可以先后存在多次执行 | 用户(可选;不提供则该任务没有业务级幂等) |
| **ExecutionID** | **不可复用**的执行身份。一次执行尝试的永久标识 | 创建即定,永不复用、永不释放;终态后其记录不可变 | 系统(ulid),**用户不可指定** |
| **ReplayOf** | 新执行由哪次执行重放而来的指针(ExecutionID → ExecutionID) | 创建时写入,不可变;不重放则为空 | 系统(Replay 时) |
| **DependsOn** | 依赖引用。**永远引用 ExecutionID**,不引用 BusinessKey | 创建时写入,不可变 | 用户(提交时给出父 ExecutionID) |

一图看关系:

```text
BusinessKey "daily-report:2026-07-16"
    │ 1 ─── N(同一业务键下的执行历史,按 ReplayOf 排成一条链)
    ▼
Execution E1(failed) ◄──ReplayOf── Execution E2(completed)
```

依赖引用的示例(父成功、后被重放的场景):

```text
BusinessKey "ocr:doc-42"
    └─ E1(completed,Result=r1)◄──ReplayOf── E2(completed,Result=r2)

C(DependsOn=[E1]):无论 E2 何时出现、结果是什么,
C.DependsOn 永远指向 E1,Get(E1) 永远返回 r1。
```

## 3. 语义规则(裁决逐条展开)

1. **现有 `Task.ID` 逐步转成 ExecutionID**。迁移方向:今天的 `Task.ID` 语义上就是 ExecutionID(自动生成的 ulid 天然满足"不可复用");问题只出在 `WithID` 允许用户把业务键写进这个字段。
2. **`WithID` 不再承担两种职责**。用户要业务幂等,走 BusinessKey(新概念);要引用一次执行,用系统生成的 ExecutionID。过渡期 `WithID` 的兼容策略是 spec 的事,模型只规定终态:**ExecutionID 用户不可指定**。
3. **Retry 只处理同一 execution 的失败重试**。即现有的 Attempts/退避机制——同一个 ExecutionID 内部的多次尝试,不产生新身份。"Retry"这个词保留给这一层,**不做"终态任务原地重跑"的 API**(裁决:e 暂缓)。
4. **Replay 创建新的 execution,记录 ReplayOf**。重放 = 新身份 + 指回旧身份,不是改写旧身份。
5. **旧 execution 的 Result、计数、错误永不被新执行覆盖**。终态即不可变——这是整个模型的地基,审计与依赖溯源都靠它。
6. **子任务永远绑定旧 execution,不自动跟随 replay**。E1 被重放出 E2 后,`DependsOn=[E1]` 的子任务看到的还是 E1 与它的结果;想消费 E2 的结果,提交新的子任务引用 E2。

## 4. Submit 幂等与 Replay 的规则(模型推荐,待审)

裁决没有直接给 Submit 的去重规则,以下是按"契约 2 精神 + 显式重放"推导的推荐,**属本文档需要审查确认的部分**:

- **Submit(带 BusinessKey)**:该键下**存在任何 execution(不论状态)→ 拒绝**(沿用 `ErrTaskExists` 语义)。理由:保住契约 2"同键二次提交必拒"的强幂等——cron 双触发、网络重试都被挡住;"失败后想再跑"不走 Submit,走显式 Replay。
- **Submit(不带 BusinessKey)**:无业务级幂等,每次都是新 execution(等同今天不用 `WithID` 的行为)。
- **Replay(按 BusinessKey 或 ExecutionID 指定)**:前置条件三条——①目标 execution 已终态;②该 BusinessKey 下**没有非终态 execution**(一个业务键同一时刻最多一次在途执行);③**目标尚未被重放过**(不存在 `ReplayOf` 指向它的 execution,即目标必须是链尾)。按 BusinessKey 指定时天然作用于链尾(最新 execution);按 ExecutionID 指定非链尾 execution → 拒绝。效果——创建新 execution(新 ExecutionID、计数清零、Payload 复制自旧执行或允许覆盖,待 spec),写 `ReplayOf`,进正常调度。
- **为什么要③(链不分叉)**:没有它,E1(failed)←E2(completed) 都终态后再对 E1 做 Replay,前置条件①②全过,产出 E3(ReplayOf=E1),历史从链变成树,"历史链可枚举且有序"就不成立了。有了③,无 BusinessKey 的执行(见第 7 节问题 #4)也自动安全:任何重放只能接在链尾、且链尾必须终态,所以**每条链同一时刻至多一个非终态执行**,不依赖 BusinessKey 不变式。
- **Replay 允许重放什么终态**:`failed`/`canceled` 无争议;**`completed` 是否允许重放**是待裁决点(允许=支持"重新生成报告",风险=误触发重复计费;倾向允许但要求显式参数,见第 7 节问题清单)。

## 5. 状态与生命周期

**Execution 的状态机不变**:七状态(blocked/pending/running/retrying/completed/failed/canceled)与现有 `canTransition` 完全沿用——本模型不加状态、不加流转,只是把"谁可以再次出现在这个状态机里"从 ID 层面挪到了 BusinessKey 层面。

**BusinessKey 视角的生命周期**:

```text
(不存在)──Submit──► 有且仅有一个在途 execution ──终态──► 全部历史均终态
                          ▲                                │
                          └────────── Replay ──────────────┘
                       (新 execution 入链,键回到"有在途"状态)
```

不变式(五后端都要维护),两条:

1. **同一 BusinessKey 下,非终态 execution 数 ≤ 1**。
2. **每个 execution 至多被重放一次**(至多一个 execution 的 `ReplayOf` 指向它)——历史因此是链不是树;配合"Replay 目标必须终态",无 BusinessKey 的链同样得到"链尾至多一个非终态"。

## 6. 对既有系统的影响面(概览,spec 阶段展开)

| 受影响处 | 变化 |
|---|---|
| `Task` 结构 | 新增 BusinessKey(可空)、ReplayOf(可空)两个字段;`ID` 语义收紧为 ExecutionID |
| brokertest 契约 2 | 重写:幂等判定从"任务 ID"改到"BusinessKey 上最多一次提交";`WithID` 相关用例随兼容策略调整 |
| 存储(五后端) | 各加两列/两字段 + BusinessKey 索引(幂等判定要按键查非终态 execution,唯一性约束的实现方式各介质自定) |
| `WithID` | 职责拆分,兼容策略待 spec(候选:废弃并引导到 BusinessKey / 保留为 BusinessKey 的别名) |
| cron 配方(README) | 重写:`Submit + BusinessKey` 去重;失败后 `Replay` 重跑 |
| `List`/`Filter` 查询 | **新增按 BusinessKey 的查询/过滤能力**(现有 `Filter` 没有该字段)。cron 使用方必须能回答"这个键现在什么状态、最新执行是谁"才能决定要不要 Replay;第 7 节原型断言④"历史链可枚举且有序"也隐含要求它 |
| `DependsOn` 校验 | 不变("父必须已存在"),只是明确校验对象是 ExecutionID |

**明确不变的**:Broker 接口的 15 个方法签名(Replay 若落地是否进 Broker 接口属 spec 决策)、状态机、三计数分工、租约/心跳/reaper、依赖唤醒的原子性契约。

## 7. 原型验证点与待定问题

**原型(裁决第 4 步之一,先于 spec)**:构造 `E1(completed, r1) ← C(DependsOn=[E1], 已消费 r1)`,对 E1 做 Replay 得 E2(结果 r2)。断言:①`Get(E1)` 仍返回 r1、计数与错误原封不动;②C 及任何后续按 `C.DependsOn[0]` 的查询看到的是 E1 不是 E2;③E2 有 `ReplayOf=E1`;④BusinessKey 下的历史链可枚举且有序;⑤E2 存在后再对 E1 做 Replay 被拒绝(链不分叉)。原型可先在 memorybroker 上做,验证的是模型不是存储。

**spec 前必须回答的问题清单**:

1. ~~Submit 对"键下已存在 execution"是一律拒绝,还是"最新为终态时自动 Replay"?~~ **✅ 评审确认(2026-07-16):一律拒绝**。自动重放会让 Submit 的语义变成"有时新建、有时重放",重复触发直接变重复执行,把契约 2 的强幂等拆了。
2. ~~Replay 是否允许对 `completed` 执行?~~ **✅ 评审确认(2026-07-16):允许,且必须显式参数**(调用方必须写明允许重放已完成的执行)。"重新生成报告"是真实需求,误触发重复计费的风险由显式参数挡住,与"重跑必须显式"的模型精神一致。
3. Replay 的 Payload:复制旧的,还是允许提交新 Payload?(倾向:默认复制,允许覆盖)
4. 无 BusinessKey 的 execution 是否允许 Replay?(倾向允许——ReplayOf 不依赖业务键;链不分叉不变式已保证无键链的安全,见第 4、5 节)
5. `WithID` 的兼容路径与弃用节奏。**评审提醒:"保留为 BusinessKey 别名"不是低破坏选项**——今天 `WithID("my-key")` 之后可以 `Get("my-key")`、`DependsOn("my-key")`,ID 收紧为系统生成后这两个用法在别名方案下照样全断,两个候选的破坏面其实差不多。
6. 两条不变式("键下 ≤1 非终态"、"每个 execution 至多被重放一次")在各介质的实现(唯一部分索引 / 条件唯一约束 / Lua 校验)与并发竞态(并发 Submit 同键、并发 Replay 同目标)。
7. Submit 拒绝时的错误携带信息:现有 `ErrTaskExists` 是裸哨兵错误,调用方拿不到已存在执行的 ExecutionID 与状态,要不要在错误里带出来(否则调用方得按 BusinessKey 查询绕一圈才能决定 Replay)。
