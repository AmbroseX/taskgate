## 关键裁决备查(research.md + 实现沿途)

- **单段 Lua 认领,消灭中转窗口**:放弃方案 v5 原文的"BLMOVE 中转 + reaper 扫中转区"两步,认领 = claim.lua 一段脚本原子完成(搬到期 delayed → LPOP 校验循环 → 写租约/令牌/inflight/计数)。"任务已离队却无租约"的崩溃态由构造直接不可达(FR-003),且 BLMOVE 真阻塞挂真时间会让 fakeclock 破功;Dequeue 是 Go 侧循环,三个唤醒源(同进程换代式唤醒信号 / 注入 clock 的 min(100ms, 最近到期时刻) / ctx)
- **时间与随机全部 ARGV 注入**:脚本内禁 TIME/math.random,now 与 ulid 由 Go 生成传入——fakeclock 在 miniredis 上直接有效,脚本纯确定性,真 Redis 复制/重放安全
- **能力接口接分布式限流**:限流进不了 Broker 的最小公倍数(memory/sqlite 没有跨进程介质),拆成 LimiterProvider 可选能力接口;scheduler 断言的是接口不是具体后端类型,上层零 import 后端包,不违宪法 II.2
- **传播单段脚本收敛**:finish.lua 五个 op 共用,工作队列在同一段脚本内收敛整棵子树(唤醒/连锁取消),"父完成子未唤醒"不可观测;reap.lua 保留 blocked 防御修复与"过期+cancel_requested → canceled"(取消不因崩溃丢失)
- **续期丢槽重占**:信号量续期用无条件 ZADD(upsert)而非 ZADD XX——续期被拖过 TTL 槽被清掉时等于"静默重占"。取舍:槽表达"本进程确实还在跑一个任务",放弃槽位会导致别的进程占走空位后全局实际并发永久超限;重占最多短暂超限一拍,靠对方槽自然过期收敛,两害取轻
- **断连冷却重试,错误绝不折叠**:网络/IO 错误原样透传,绝不折叠成 ErrLeaseLost/ErrTaskNotFound(一次网络抖动不许丢结果);AcquireSlot/WaitToken 对 Redis 抖动也不上抛,冷却后重试(claimLoop 把限流错误当"停止认领",断连期间消费循环必须活着);ctx 取消时 go-redis 漏出的网络形态错误统一翻译回 ctx.Err()(sqlite SQLITE_INTERRUPT 同款教训)
- **redis_rate 真时钟豁免**:GCRA 的 Lua 用 Redis 服务器时间,属 M1 已注明的"RPS 走真时钟"唯一豁免(spec FR-018 例外),与其余脚本"时间全注入"的铁律不冲突——限速要的本来就是物理时间
- **观测客户端关自动重试**(实现沿途):多进程测试的 INCR/DECR 观测计数不幂等,go-redis 默认自动重试在高负载下会把计数悄悄写两遍制造假超限,观测连接统一 MaxRetries=-1;fakeclock 新增 TickerCount 供测试等"异步注册的续期 ticker 挂上假时钟"再拨钟,消除拨钟先于注册导致的滴答永久丢失
- **DequeueAck 基准裸调 Broker**:不经 scheduler/limiter(等价"限流关掉测框架开销"),Enqueue→Dequeue→Ack 一整圈为一个 op,量的是后端三次核心调用的纯开销

## 已知限制与 M3 待办(对照方案 v5 第 11 节 M3 行)

- **不支持 Redis Cluster**:脚本用 ARGV 前缀自行拼键、不带 hash tag,面向单实例/主从/哨兵;Cluster 支持需键设计重做(hash tag 按队列分片 + 跨槽脚本拆分)
- **redis_rate 真时钟豁免**继续存在:RPS 限速用 Redis 服务器时间,fakeclock 管不到,相关测试只能真时钟短窗口(容差 ±2)
- **跨进程 Dequeue 100ms 轮询**:别的进程写入靠兜底轮询发现(同进程有唤醒信号即时响应);M3 若要更低延迟可叠加 BLPOP 通知键快路径
- **限流键与任务键同实例**:flushdb 级故障两者同生共死(诚实的取舍,spec Edge Case 已注明)
- **空非 nil 的 `[]byte{}` Result 经 redis 往返变 nil**:hash 里空串与"没有"不可分,取回按 nil 还原,与 memory(原样保留空切片)有细微差别;brokertest 未约束该边缘,调用方不应依赖"空切片 vs nil"的区分
- **claim 的 delayed 搬运与 reap 的 blocked 扫描无分页上限**:单脚本一口气处理 O(N)(N=到期 delayed 数 / blocked 总数),目标场景(单实例、万级任务)可接受;若 blocked 常态上十万需引入分批
- M3 功能待办:webhook 通知等重语义、handler 手动续租 API(长任务超过 LeaseTTL 目前只能靠心跳自动续)、List 分页(当前全量取回再截 Limit)、任务优先级、真实网关冒烟档(L5)
- 回执/reaper 用 Background ctx,后端假死时 Shutdown 可能卡住(M1 遗留 backlog,redis 后端有 limiterOpTimeout 兜底,Broker 调用侧仍未设上限)
