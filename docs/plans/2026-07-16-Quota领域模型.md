性质:**领域模型文档,不改代码、不定接口签名**。接口形态(独立能力接口还是别的)是 spec 的事,模型只定语义与约束。

## 1. 裁决定型(六件事,全部已拍板)

| 维度 | 裁决 |
|---|---|
| **配额单位** | 启动一次 handler(明确排除 token 计量——那需要"执行后上报用量"模型,属另一个功能) |
| **窗口** | 固定时长窗口,对齐窗口起点公式 `windowStart = now / period * period`;不支持自然日/自然月(第二版再议,需 IANA 时区 + DST 定义) |
| **时间** | **后端共享介质的服务端时间**(见第 4 节);生产窗口键不得依赖应用进程本地时间;`clock.Now()` 只用于测试 |
| **保证** | **硬配额**:窗口内启动的 handler 次数**绝不超过** QuotaLimit。接受代价:预留了但没真正执行,额度照样消耗(保守损耗) |
| **作用范围** | `queue + quota key`。默认 quota key = 队列名;多个队列打同一个网关额度时,配同一个 quota key 共享预算 |
| **降级** | **不存在静默降级**。后端不支持共享计数 → 装配时(`New()`)显式拒绝该配置报错。宁可不启动,不给假保护 |

**命名后备(裁决原文记录)**:若团队最终不能接受硬配额的保守损耗,这个功能**不得叫 quota**,改名 `AdmissionBudget` 并公开写明形式化的超发上界。"叫 quota 但会超发"的中间态不存在。

## 2. 核心对象:QuotaReservation

一次"额度预留"。它是配额语义的原子单位,生命周期:

```text
            后端原子操作:检查窗口 + 扣减/预留
                        │
   (无额度:不预留,认领暂停到下个窗口)
                        ▼
                    reserved ──认领成功,handler 启动──► consumed(正常路径)
                        │
                        ├──认领扑空/出错,尽力退还──► released(补偿,尽力而为)
                        │
                        └──进程在中间崩溃──► leaked(泄漏 = 视同 consumed,保守方向)
```

规则:

1. **预留在认领之前**,由后端在共享介质里**一个原子操作**完成"检查当前窗口余额 + 扣减"(redis 单条 Lua;SQL 单条 `UPDATE ... WHERE used < limit` 或 `INSERT ... ON CONFLICT` 带条件)。检查与扣减之间没有窗口,这是"硬"的来源。
2. **consumed**:认领到任务、handler 启动,预留转为消耗。这就是配额单位的落点。
3. **released**:预留了但认领扑空(队列空)或认领出错,**尽力**退还(再一个原子操作把计数减回)。退还失败不重试到死——失败就当 leaked。
4. **leaked**:进程在"预留成功、认领未完成"之间崩溃,这份额度没人退还,当作已消耗。**方向永远保守**:任何故障都只会少放行、不会多放行——这是硬配额的定义,写进合同。
5. 预留没有 TTL、没有后台回收:回收机制会引入"回收和迟到的认领并发"这类竞态,且泄漏量有自然上界(每个认领循环同时至多挂一个预留),不值得为它开新机制。窗口切换后旧窗口的泄漏自动失效(新窗口新计数)。

**备选路线(记录,未选)**:后端提供专门的 admission/claim 原子能力("Dequeue + 扣配额"一步完成),无白烧无泄漏,但要动 Broker 接口、五后端都得有对应原子形态,代价大;若基准测试发现预留模式的空轮询白烧不可接受,再回头评估它。

## 3. 耗尽行为(沿用调研方案 5.3 的既有结论)

- 配额耗尽 → 该 quota key 下的队列**停止认领**,等下个窗口;**不占 worker 槽**、不把任务认领出来再拒绝
- 已在跑的任务不受影响(配额管"放行",不管"中断")
- 耗尽**不是错误**:不进 `Throttled` 计数、不进 `failed`,任务老实待在 `pending`
- 与 `ErrThrottled` 的分工不变:配额是"我们自己知道不该发"(任务没发出去),`ErrThrottled` 是"网关说它忙"(任务已发出去)

## 4. 时间:每个介质的"服务端时间"是什么

裁决要求窗口键用共享介质的服务端时间,逐介质落点:

| 介质 | 服务端时间来源 | 说明 |
|---|---|---|
| redis | `TIME` 命令,在同一条 Lua 里取时并计算窗口键 | 所有客户端看同一个钟,偏差问题消失 |
| PG / MySQL | `now()` / `NOW(3)`,在同一条扣减语句里取时 | 同上 |
| sqlite | 本机时钟 | 共享介质是本机文件,能打开它的进程都在同一台机器、同一个时钟源——"本机钟"就是这个介质的服务端钟,语义等价 |
| memory | 进程时钟 | 介质就是本进程内存,单进程语义下无偏差问题 |

**与 fakeclock 铁律的调和**:"所有时间由注入 clock 提供"这条测试铁律**保留,但边界收窄为测试**——生产路径的窗口计算用介质时间;测试路径通过介质侧注入覆盖(redis 有 Lua ARGV 注时的现成先例;SQL 侧在测试构造时注入"时间覆盖"参数,具体缝的形态属 spec)。两条都写进合同:生产不信任节点钟,测试不真 sleep。

## 5. 故障语义(fail-closed)

- **共享介质不可达**(断连/超时):该 quota key 下**暂停认领**,按退避重试预留;**绝不**退回进程内计数放行。配额宁可停,不可假
- **装配期**:后端不支持共享计数却配了配额 → `New()` 报错,错误信息写明哪个队列哪个配置不被该后端支持
- **错误可见性**:介质不可达导致的认领暂停要可观测(错误日志/计数),不能表现为"队列莫名不动了"

## 6. 与现有限流的关系

三个维度正交,谁也不替代谁:

| 维度 | 管什么 | 现状 |
|---|---|---|
| slot(Workers) | 同时在跑多少 | 已有 |
| rate(RPS/Burst) | 每秒新启动多少 | 已有 |
| **quota(本模型)** | **每个窗口累计启动多少** | 新增 |

接口形态待 spec,但模型给两条硬约束:①**不得沿用现有 `LimiterProvider` 的静默退化模式**(它是"后端不实现就悄悄退回进程内",与本模型第 1 节"不存在静默降级"直接冲突);②不得把配额塞进 `QueueConfig` 之外的通用钩子(调研方案 8.5 的结论)。接口候选的比较记录在调研方案 5.4,模型裁决后大概率收敛到"独立的配额能力 + 装配期强校验"一类形态,由 spec 定。

## 7. 原型验证点与基准(先于 spec)

**原型(裁决第 4 步之二)**:两个进程共享同一介质(可用 pg 或 sqlite 同文件),同一 quota key,`QuotaLimit=N`、短窗口。验证:①两进程合计每窗口启动 handler 次数 ≤ N(含窗口边界前后——服务端时间下不许出现双窗口放量);②窗口切换后额度恢复;③一个进程 kill -9 在"预留后认领前",总放行只少不多;④介质断连期间零放行,恢复后继续。

**基准(裁决明确要求"性能是否足够靠基准测试决定")**:预留是认领热路径上的一次原子写,且同一 quota key 的所有消费者**争同一行/同一键**。基准至少覆盖:pg/mysql 单行热点争用(多连接并发 UPDATE)、sqlite 单写者串行下的吞吐、redis Lua。产出对比数:有配额 vs 无配额的认领吞吐损耗百分比,写进 spec 的已知代价。

## 8. spec 前必须回答的问题清单

1. 预留-退还(released)的原子退还语句在各介质的具体形态;退还与窗口切换赛跑时的行为(旧窗口键退还落空 = 无害,确认即可)。
2. 空队列长轮询下的白烧节奏:认领循环在"预留成功但队列空"之后要不要退避(否则每次轮询烧一个预留、尽力退还,失败即漏)。**评审给的缓解候选**:预留前先查 `QueueLen(queue) > 0` 再预留(Broker 已有此方法)——有竞态所以只是启发式不是保证,偶尔白烧照走尽力退还,但把"空队列每次轮询烧一个预留"的常态消掉;再配一个空扑后的退避。**先例**:认领循环"先拿令牌再 Dequeue"本就接受空闲期预烧至多 1 个令牌(scheduler.go 的"顺序写死"处,M1 spec SC-001 的 ±1 容差),预留白烧是同一个代价形状,本项目有先例可循。
3. quota key 的配置面:字段名、默认值(=队列名)、多队列共享的校验(同 key 不同 QuotaLimit 要报错)。
4. 测试时间缝的形态:redis 沿 ARGV;SQL 侧用什么方式注入覆盖时间(连接参数/构造选项/测试专用表)。
5. 观测口:配额余额、本窗口已用、泄漏估计要不要暴露进 `Overview`。
6. 若基准测试不过关(热点行争用拖垮认领),是否回退到"备选路线"(admission/claim 原子接口)——触发条件先写死(如吞吐损耗 > X%)。
7. 预留在既有认领顺序链中的位置:现在的顺序写死是"先占并发槽 → 等 RPS 令牌 → Dequeue"(scheduler.go),预留插在哪一环?若排在占槽之后,配额耗尽的那一刻循环手里正握着一个槽,得先释放槽再暂停认领——语义上与第 3 节"不占 worker 槽"不矛盾(瞬时持有不算占),但衔接细节要在 spec 写清楚,别到实现时才发现和现有顺序打架。
