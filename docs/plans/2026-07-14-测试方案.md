# taskgate 测试方案

> 配套设计方案：`2026-07-14-任务排队限流库taskgate方案.md`（v4）
> 原则：所有自动化测试**离线可跑、结果确定**——LLM/OCR 一律用 httptest 起 mock 服务，真实网关只留手动冒烟档

## 0. 分层总览

```
L5  真实网关冒烟（手动，build tag 隔离，不进 CI）
L4  仿真 E2E：mock LLM/OCR 网关 + 完整流水线 + 故障注入
L3  集成测试：Client+Scheduler+Broker 全链路场景
L2  brokertest 合规套件：三后端跑同一套行为契约
L1  单元测试：limiter / 退避 / 配置校验 / 依赖计数等纯逻辑
```

L1~L4 全部进 CI（`go test ./... -race`），下层越多越快，上层只保关键路径。

## 1. L1 单元测试（纯逻辑，无 IO）

| 对象 | 用例 |
|---|---|
| 限流器 | RPS=10 时 1 秒放行 10±1 个；Burst 生效；Workers=2 时第 3 个必须等前面归还槽位 |
| 退避计算 | 指数曲线 `min(2^n×1s,10min)`；抖动在 ±20% 内；ErrThrottled 用 RetryAfter 不用曲线 |
| 配置校验 | Workers=0 / RPS<0 / Handle 引用不存在的 Queue → New 返回 error 不 panic |
| 任务选项 | WithID / Delay / RunAt / DependsOn / MaxRetry 组合的合法性 |
| 依赖计数 | pending_parents 减到 0 才就绪；重复减不会变负数 |
| 状态机 | 非法流转（completed→running 等）一律拒绝，表驱动全枚举 |

## 2. L2 brokertest 合规套件（三后端同一套题）

`brokertest.Run(t, func() Broker {...})`，memory / sqlite / redis(miniredis) 各自一行接入。契约清单：

1. **Enqueue/Get 往返**：写进去读出来字段一致（含 Payload/RunAt/DependsOn）
2. **幂等 ID**：重复 Enqueue 同 ID → ErrTaskExists，原任务不被覆盖
3. **认领互斥**：同一任务 100 次并发 Dequeue 只被认领一次
4. **阻塞语义**：空队列 Dequeue 阻塞；Submit 后返回；ctx 取消立即退出（只测语义不测轮询/BLMOVE 手段）
5. **延迟任务**：RunAt 未到不出队，到了才出队
6. **Ack/Fail 流转**：Ack 写 Result 置 completed；Fail 带 retryAt 置 retrying、耗尽置 failed
7. **租约与回收**：认领后不 Ack，租约过期 ReapExpired 捞回 pending，LeaseLost+1；Heartbeat 能续租
8. **依赖唤醒原子性**：父 Ack 后子必然从 blocked 变 pending（无中间态可观测到"父完成但子未唤醒"且不再有后续动作）
9. **连锁取消**：父 failed → FailFast 子 canceled → 孙 canceled（链式，允许短暂延迟，最终一致）
10. **Cancel 各状态**：blocked/pending/retrying 直接 canceled；对终态任务 Cancel 报错
11. **Counts 一致性**：任意一串操作后 Counts 与逐个 Get 汇总一致（memory 后端同样必须过）
12. **List 过滤**：按 Type/Status/Queue 过滤正确

redis 后端额外加真 Redis 档（`TASKGATE_REDIS_ADDR` 有值才跑）：miniredis 保 CI，真 Redis 保 Lua 脚本兼容性。

## 3. L3 集成测试（New→Handle→Run 全链路）

统一用 memory/sqlite 后端 + 短租约（100ms 级）加速：

| 场景 | 步骤与断言 |
|---|---|
| 提交→执行→取结果 | Submit → Wait 拿到 Result；Get 看到完整时间戳链 CreatedAt≤StartedAt≤FinishedAt |
| Wait 超时 | handler 睡 1s，Wait ctx 500ms → 返回 ctx 错误，任务照常跑完 |
| 重试链路 | handler 前 2 次返回 error 第 3 次成功 → Attempts=3、最终 completed；退避间隔递增 |
| ErrThrottled | handler 返回 ErrThrottled{1s} 3 次 → Attempts 不涨、RunAt 每次 +1s、最终成功 |
| ErrSkipRetry | 返回即 failed，不重试 |
| 毒任务 | 模拟 worker 不 Ack（handler 卡死+杀 goroutine）→ LeaseLost 到 3 → failed |
| 取消 running | Cancel 后 handler 的 ctx 在租约心跳周期内被 cancel；状态 canceled |
| 三级流水线 | A→B→C（DependsOn 串联），A 完成唤醒 B、B 完成唤醒 C；B 配 FailFast 时 A 失败 → B、C 连锁 canceled |
| fan-in | C 依赖 A+B 两个父，只有两个都完成才跑；IgnoreParentFailure 时 A 失败 C 照跑 |
| 提交时父已完成 | 先跑完 A 再提交 DependsOn(A) 的 B → B 直接 pending 不卡 blocked |
| 限流隔离 | 两个队列 {W:1,RPS:1} 和 {W:8}，灌满慢队列不影响快队列吞吐 |
| Shutdown 正常 | 在跑 3 个任务时 Shutdown(5s) → 3 个都 completed 才返回，期间不再出队 |
| Shutdown 超时 | handler 睡 10s，Shutdown(500ms) → 返回超时错误、handler ctx 被 cancel、任务留在租约里事后可回收 |
| 统计 | 造一批已知分布的任务 → Overview 矩阵、Stats 水位、List 过滤三者对得上 |

## 4. L4 仿真 E2E：mock LLM / OCR 网关 + 故障注入

用 `httptest.Server` 造两个假网关，**把生产踩过的坑做成注入开关**：

```go
mockLLM := NewMockGateway(
    Latency(200*time.Millisecond),
    BusyAfterConcurrency(2),   // 复刻讯飞：并发>2 返回 busy（错误还藏在 200 的 SSE 体里）
    FailRate(0.1),             // 10% 随机 500
)
mockOCR := NewMockGateway(
    Latency(2*time.Second),
    CrashAfterConcurrency(4),  // 复刻 feedoc：4 路并发直接断连不自愈
)
```

核心用例：

1. **LLM 批量抽取**：100 个任务打 mockLLM，队列 {Workers:2}——断言 mock 端观测到的最大并发≤2、无一个 busy 触发（限流真的挡住了）；故意改成 {Workers:5} 则 busy 全部走 ErrThrottled 延后重排、零任务 failed
2. **OCR 灌库**：20 个 PDF 任务打 mockOCR，队列 {Workers:2}——mock 不崩、全部 completed；{Workers:4} 时断连错误走普通重试、退避后在并发降下来时补完
3. **完整流水线**：`ocr → llm-extract → llm-score` 三类型三队列各自限流，10 份文档并行灌入 → 30 个任务全 completed，且 score 任务的 handler 里能 Get 到 extract 的 Result
4. **流水线中途取消**：文档跑到 ocr 完成、extract 排队时 Cancel extract → score 连锁 canceled，ocr 保持 completed
5. **SSE 藏错误**：mockLLM 返回 200 但 body 是错误事件 → handler 判定后返回 ErrThrottled，验证"HTTP 状态码骗人"场景的重排路径

## 5. 故障与竞态专项

| 项 | 手段 | 断言 |
|---|---|---|
| 数据竞争 | 全量 `-race` 跑 L1~L4 | 零 data race |
| 崩溃恢复（单机） | 子进程跑 worker 处理到一半 `kill -9`，主进程重开同一个 sqlite 文件 | 任务被 reaper 回收重跑，最终 completed，LeaseLost=1 |
| 唤醒中途崩 | 用注入点让"父 Ack 事务提交前"崩 | 重启后父仍 running 租约过期→回收重跑→子正常唤醒（不丢唤醒） |
| 双进程抢任务 | 两个进程共用 sqlite / 共用真 Redis，灌 1000 个任务 | 每个任务恰好执行一次（handler 里往结果表写执行记录数） |
| 跨进程流水线 | 进程 A 只跑 ocr 队列、进程 B 只跑 llm 队列 | 依赖唤醒跨进程生效 |
| Redis 断连 | mock 网关正常、中途重启 miniredis | worker 报错重连后继续消费，不丢任务 |
| 时钟相关 | 租约/退避全部走可注入的 clock 接口 | 测试不用真 sleep 长时间，CI 快且稳 |

## 6. 性能基准（M2 完成后跑，记录基线防退化）

- `BenchmarkEnqueue`：单协程/32 协程并发入队（sqlite vs redis）
- `BenchmarkDequeueAck`：空转吞吐（限流关掉，测框架开销）
- `BenchmarkPipeline`：三级依赖链每秒能推进多少条
- 目标不定死数字，首次跑出基线写回本节，后续 PR 对比 ±20% 内

## 8. 目录与运行方式

```
taskgate/
├── limiter_test.go 等            # L1，跟随源码
├── brokertest/                   # L2 套件本体
├── memorybroker/broker_test.go   # 一行接入 brokertest
├── sqlitebroker/broker_test.go
├── redisbroker/broker_test.go
├── integration_test.go           # L3
├── e2e/
│   ├── mockgw/                   # mock LLM/OCR 网关（故障注入开关）
│   ├── pipeline_test.go          # L4
│   ├── crash_test.go             # L5 专项（子进程 kill -9）
│   └── realgw_test.go            # //go:build realgw
```

```bash
go test ./... -race                # CI 全量（L1~L4 + 专项）
go test ./... -race -short         # 跳过慢用例（崩溃恢复、双进程）
TASKGATE_REDIS_ADDR=127.0.0.1:6379 go test ./redisbroker/...   # 真 Redis 档
go test -tags realgw ./e2e/ -run RealGW                        # 手动冒烟
go test -bench . -benchmem ./...   # 基准
```

## 9. 覆盖率要求与里程碑对照

- 语句覆盖率目标：核心包（scheduler/limiter/deps）≥ 85%，broker 实现 ≥ 80%；不为凑数写无断言测试
- **M1 完成的定义** = L1 全绿 + brokertest 在 memory/sqlite 全绿 + L3 全场景 + 崩溃恢复/唤醒中途崩两个专项
- **M2 完成的定义** = brokertest 在 miniredis+真 Redis 全绿 + 双进程/跨进程流水线专项 + 基准基线入档
- **M3 试点前** = L4 全绿 + 真实网关冒烟跑通一轮
