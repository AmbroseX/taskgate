## 关键裁决备查(research.md + 实现沿途)

- **RenewLease 走 ctx 注入闭包,不改 Handler 签名**:Handler 签名 M1 已定型,加参数破坏全部现有代码;Task 挂 Renew() 方法要持 broker 引用,污染序列化边界。非导出 ctx key 保证只有 scheduler 能注入,外部伪造不了;非任务 ctx 返回导出的 ErrNoTask 不 panic
- **续租闭包复用心跳的三优先级错误语义**:不造第二套租约状态机——ErrTaskCanceled 先 cancel 任务 ctx 再原样返回、ErrLeaseLost/ErrTaskNotFound 置"租约已丢"标记+cancel+返回 ErrLeaseLost、网络错误原样;"租约已丢"标记复用后续回执判定,第一跑作废结果自动丢弃
- **手动档不留半自动后门**:ManualHeartbeat=true 时不保留"只查取消不续租"的低频轮询(语义混浊,排除);跨进程 Cancel 由下一次 RenewLease 发现,一直不续则租约过期兜底——"handler 对自己的租约负全责",写进 godoc 与 README
- **分页选 Offset 不选游标**:排序键 (CreatedAt, ID) 保证自定义 ID(幂等去重)场景下仍是全序;游标分页三后端实现成本高,YAGNI,M4 再议;翻页弱一致("未变动的任务不丢不重")写进合同文档
- **redis List 不加 zset 二级索引**:List 本来就要 HGETALL 全字段,内存排序不改变渐进复杂度;为一个管理查询在每条流转多维护第 4 个结构不值,O(候选集) 写进已知限制
- **mockgw 判定顺序 = 断连 → busy → 500 → 延迟**:断连/busy/500 是网关过载时的快速拒绝,不吃 Latency;只有正常请求睡出处理耗时——这样 {W:2} 档并发才真实地被撑起来可观测,busy 档的触发也即时确定
- **busy 藏在 HTTP 200 里是刻意的**:handler 只有解析 body 才能发现错误(测试里 handler 判定 `"error":"busy"` 后返回 ErrThrottled),完整复刻"HTTP 状态码骗人"的真实网关;用例⑤直接复用该形态
- **L4 单后端(memory)跑五用例**:L4 测的是"限流×故障×流水线"的组合行为,后端矩阵 L2(18 条契约)/L3(集成场景)已盖三后端;单后端控制 CI 时长(e2e 包 -race 全程 ~20s)
- **档二 ThrottledMax 抬到 1000**(实现沿途):busy 是快速拒绝,5 worker 空转时判定频率高(实测平均每任务 ~25 次),默认 100 封顶在满负载 CI 下有小概率把倒霉任务打进 failed,属测试参数不是行为改动
- **断连用例 {Workers:6} 而不是 spec 原文的 {Workers:4}**(实现沿途):断连条件是并发 >4,{W:4} 永远打不破红线,tasks.md 已注明"自行调参让断连真的发生";{W:6} 必然触发(实测 CrashCount≈24)
