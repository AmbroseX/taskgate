## 1. Identity 原型(memorybroker,真调度)

做法:Task 还没有 BusinessKey/ReplayOf 字段,原型用侧车 map + 一把互斥锁
包一层 taskgate 公共 API(锁模拟正式实现里"唯一约束 + 原子校验"的职责,
单进程内语义等价),任务真跑真出结果。

| 断言(模型第 7 节) | 结果 |
|---|---|
| ① Replay 后 `Get(E1)` 仍返回 r1,Result/Attempts/LastError/Status/FinishedAt 逐字段不变 | ✅ |
| ② C.DependsOn[0] 永远指向 E1;replay 后新提交的子任务引用 E1 看到的也是 r1 不是 r2 | ✅ |
| ③ E2 有 ReplayOf=E1 | ✅ |
| ④ History(key) = [E1, E2, E3] 可枚举且有序 | ✅ |
| ⑤ E2 存在后再对 E1 做 Replay 被拒绝(链不分叉) | ✅ |

规则性拒绝(评审补充项)也全部验证:

- 同键二次 Submit 一律拒绝(`ErrTaskExists`,评审确认 #1);
- 重放 completed 不带 AllowCompleted 被拒绝(评审确认 #2);
- 键下有在途 execution 时 Replay 被拒绝(前置条件①/②);
- 按键 Replay 天然打在链尾(E3 的 ReplayOf 是 E2 不是 E1);
- 无键执行可以 Replay,且链不分叉不变式同样生效(不依赖 BusinessKey)。

**发现**:模型语义在现有调度/依赖唤醒机制上零阻力——不用动状态机、不用动
DependsOn 校验,五断言全靠"新身份 + 指回旧身份 + 旧记录不可变"三条就成立。
正式实现的全部难点集中在模型问题清单 #6(两条不变式在各介质的原子实现),
原型里那把互斥锁就是它的占位。

## 2. Quota 原型(sqlite 共享文件,双进程)

做法:预留 = 单条原子语句 `INSERT ... ON CONFLICT DO UPDATE SET used=used+1
WHERE used<? RETURNING win`,窗口键在语句内用 sqlite 自己的钟
(`strftime('%s','now')/period*period`)计算——"检查 + 扣减无窗口"和
"服务端时间"两条裁决都落在这一条 SQL 上。子进程模式沿用
sqlitebroker/crash_test.go 先例(exec 测试二进制 + 环境变量 + 哨兵文件)。

| 断言(模型第 7 节) | 场景 | 结果 |
|---|---|---|
| ① 每窗口合计 ≤ N(含边界) | 双进程 × 5s,period=1s,N=5 | ✅ 6 个窗口全部 =5,无一超发 |
| ② 窗口切换额度恢复 | 同上 | ✅ 4 个被完整覆盖的窗口恰好各放行 5 次 |
| ③ kill -9 只少不多 | 进程在"预留成功、认领未完成"间被杀 | ✅ 泄漏窗口恰好放行 N-1=4 次 |
| ④ 介质不可达零放行、恢复续上 | 父进程长持 `BEGIN IMMEDIATE` 1.2s 模拟 | ✅ 故障前 419 次、故障中 0 次、恢复后 497 次;子进程观测到 7 次介质错误 |

单进程单元测试另验证了:耗尽返回"非错误"(模型第 3 节)、退还的原子语句形态、
对已切走的旧窗口退还落空且无害(模型问题清单 #1 的"确认即可"项,已确认)、
不同 quota key 互不占额。

**发现**:

1. sqlite 的 `RETURNING` 让"预留成功时顺便拿到窗口起点"一条语句完成,
   放行记录的窗口归属不需要二次查询——PG/MySQL 侧 spec 时留意等价写法
   (PG 有 RETURNING;MySQL 没有,可能要 `ROW_COUNT()` + 同语句时间函数,
   属模型问题清单 #1 的展开)。
2. "长持写锁"是文件介质下模拟"介质不可达"的可靠手法(chmod/改名骗不过
   已打开的文件描述符),原型里 busy_timeout 打小(100ms)后 fail-closed
   行为完全符合模型第 5 节,值得写进 spec 的测试方案。
3. 空队列白烧在原型里没出现(原型队列永远非空),评审给的缓解候选
   (`QueueLen>0` 启发式 + 空扑退避)留给 spec,见 Quota 模型问题清单 #2。
