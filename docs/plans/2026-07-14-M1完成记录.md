- 公开 API:`New/Handle/Run/Submit/Get/Cancel/List/Stats/Overview/Wait/Shutdown`,选项 `WithID/Delay/RunAt/MaxRetry/DependsOn/IgnoreParentFailure`

- 示例 `examples/llm`(三级流水线)+ README

## 交叉审查(两个独立 agent)与修复

调度层审查:无致命。修复——untrack 按指针比对(防快速重认领误删取消句柄)、错误分类先判 ErrSkipRetry 再 ErrThrottled(防 Unwrap 穿透)、ErrUnknownType 文案统一、spec 文档三处对齐(Attempts 口径、FailFast 默认无选项、RPS 真时钟豁免)。

存储层审查:无致命。修复——ReapExpired 保留取消语义(过期+cancel_requested → canceled)、sqlite 并发 Dequeue 取消统一返回 ctx.Err()(根因:单连接下 database/sql interrupt 漏出 SQLITE_INTERRUPT)、memory Notify 快照深拷贝、空令牌契约、Notify 契约用例(第 17 条)、六个带令牌方法 requireInit、crash 测试窗口放宽、Enqueue 失败不回填 ID、fakeclock 补单测。

宪法修订 v1.1.0(MINOR):III.4 连锁取消由"仅逐层小事务"放宽为"调用返回前收敛的两种合法形态"(实现采用单事务收敛,语义更强)。

## 已知限制与 M2 待办

- redis 后端 + 分布式限流 + Lua 版依赖唤醒(接口已定型)
- RPS 令牌桶用 x/time/rate 内部真时钟,是注入 clock 的唯一豁免(spec FR-018 已注明);M2 可改 ReserveN+clock.Sleep
- 多 Type 共享队列且本进程只注册部分 handler 时,没 handler 的任务落死信(LastError 带 ErrUnknownType 文案);跨进程分工消费同一队列的场景 M2 重新设计
- sqlite Dequeue 空轮询走 IMMEDIATE 写事务探测,跨进程高频下可优化为先只读探测(backlog)
- 回执/reaper 用 Background ctx,后端假死时 Shutdown 可能卡住(本地存储风险低,backlog)
- 令牌预烧:队列空闲期至多 1 个令牌偏差(SC-001 容差内,代码有注释)
- 性能基准留 M2(测试方案第 6 节)

## 关键实现裁决备查

- handler ctx 从 Background 派生,Run 停止不连坐;回执用 Background ctx 保证落库
- 三种退出优先级:租约丢(丢弃结果)> 用户取消(FinishCanceled)> 正常返回(Ack)> Shutdown 打断(Requeue)
- Wait 轮询 50ms 走注入 clock;failed/canceled 返回 *TaskFailedError
- Delay 与 RunAt 同设时 RunAt 胜;提交时父已 failed/canceled 且 FailFast 立即 canceled(即使还有 running 父)
- 首次失败退避 backoff(0)=1s;n≥10 取 10min 封顶
