## 1. 背景与目标

调 LLM API 或跑其他耗时任务时，两个核心麻烦：

1. **并发控制**：不能一股脑打过去，网关有并发/QPS 上限（比如讯飞网关并发超 2 路就 busy），必须排队慢慢发。而且不同类型的任务要求完全不同——打分（高 QPS、短文本）、总结（低 QPS、长耗时）、评审（中等 QPS）混在一个队列里用同一套限流，要么浪费配额要么互相拖垮
2. **任务要能追踪**：异步提交后，随时查"排到没有、跑完没有、结果是啥、错在哪"，还要能取消
3. **任务之间有先后**：LLM 流水线是刚需——检索→生成→打分、草案→评审→修正。但不能因此把库做成 Temporal/Argo 那种 DAG 工作流引擎，那就不轻量了。只做"声明父任务，父任务完成自动唤醒"这一层（见第 9 节）

目标：一个**轻量、可嵌入的 Go 库**（不是独立服务），后端可插拔——单机用**本地文件**（零依赖，重启不丢），生产多实例切 **Redis**，同一套 API 一行切换。任何项目 `go get` 进来几行代码就能用。

## 2. 调研结论：为什么不直接用现成的

| 方案 | 后端 | 限流 | 本地持久化后端 | 结论 |
|---|---|---|---|---|
| [asynq](https://github.com/hibiken/asynq) | 只有 Redis | 不内建，要自己拿 x/rate 的分布式 Semaphore 拼 | ❌ | 功能最全（重试/延迟/Web UI），但强绑 Redis |
| [river](https://riverqueue.com/) | 只有 Postgres | 有并发控制 | ❌ | 事务性好，但强绑 Postgres |
| [taskq](https://github.com/vmihailenco/taskq) | Redis/SQS/内存 | 有全局限流 | ❌（内存重启即丢） | 没有"按类型独立限流"，维护活跃度一般 |
| machinery / gocelery | Redis/AMQP 等 | ❌ | ❌ | 偏重、配置繁琐 |
| pond 等 worker pool | 纯内存 | 只有并发数 | ❌ | 只管 goroutine 池，进程一退任务全丢，无状态查询 |

**空缺就是我们的定位**：轻量、可持久化（含文件后端）、**限流优先且按任务类型隔离**、状态可查。现有项目要么庞大且缺限流，要么太简单不能持久化。

诚实的替代方案：如果项目本来就重度用 Redis 且不在乎依赖，asynq + 手拼限流也能活。taskgate 的价值在"轻量嵌入 + 单机零依赖 + 类型级限流开箱即用"。

## 4. 核心概念

### 4.1 任务类型（Type）是一等公民

每个任务必带 `Type`（如 `"scoring"`、`"summarize"`、`"review"`），它决定两件事：

1. **用哪个 handler 执行**（按 Type 注册处理函数）
2. **受哪套限流约束**：默认每个 Type 自成一个队列、独享一套限流配置

**共享限额的场景**：几个 Type 打的是同一个 LLM 网关（网关限的是总并发，比如讯飞≤2），这时在 `Config.Routes` 里把多个 Type 路由到同一条队列共享限额。即：**默认 队列名=Type（类型级隔离），需要时在 Routes 里显式指定（资源级共享）**。两个需求一套机制搞定，不引入第二层限流。

**路由为什么放 Config 不挂 Handle（v5，问题①）**：任务的 Queue 字段在**入队那一刻**就要定死（Redis 后端直接 push 进 `tg:pending:{queue}`）。原方案把路由写在 `Handle("review", fn, Queue("xunfei"))` 上，但纯生产者进程不注册 handler、不调 Run，它就不知道 review 该进 xunfei 队列，任务会落进没人消费的 "review" 队列里永远没人认领。路由是生产者和消费者的**共同契约**，必须放在两边都加载的 Config 里。

### 4.2 任务模型

```go
type Task struct {
    ID        string          // 自动生成 ulid，也支持自定义（幂等去重用）
    Type      string          // 决定 handler + 默认队列
    Queue     string          // 限流单元，默认等于 Type
    Payload   json.RawMessage // 入参（见 13 节：为什么不用 interface{}）
    Status    Status
    Result    json.RawMessage
    LastError string
    Attempts  int
    MaxRetry  int
    LeaseLost int             // 租约过期被回收的次数，独立于 Attempts（见第 6 节）
    Throttled int             // ErrThrottled 重排次数，超上限进 failed，防永久 429（v5，见第 6 节）
    RunAt     time.Time       // 延迟执行/重试退避都靠它
    DependsOn []string        // 父任务 ID 列表，全部结束才会被调度（见第 9 节）
    CreatedAt, StartedAt, FinishedAt time.Time
}

const (
    Blocked   Status = "blocked"   // 等父任务完成，不进队列
    Pending   Status = "pending"
    Running   Status = "running"
    Retrying  Status = "retrying"  // 失败等待重试中（参考方案没有，见 13 节）
    Completed Status = "completed"
    Failed    Status = "failed"    // 重试次数用尽的最终失败（=死信，可查可手动重放）
    Canceled  Status = "canceled"
)
```

### 4.3 架构

```
+--------------------+      +------------------------+
|    Client API      |      |  Scheduler（调度器）    |
| Submit / Status /  |----->|  - 按队列分发           |
| Result / Cancel /  |      |  - RateLimiter（每队列）|
| List / Stats       |      |  - Worker Pool         |
+--------------------+      +-----------+------------+
                                        |
                                        v
                            +------------------------+
                            |   Broker（可插拔）      |
                            | sqlite / redis / memory|
                            +------------------------+
```

- **Client**：提交、查状态/结果、取消、列表、看各队列实时水位
- **Scheduler**：遍历各队列，哪个队列的令牌桶和并发槽有配额就拉哪个队列的任务
- **Broker**：任务存储 + 排队，接口统一，三个实现

## 5. 接口设计（用户视角）

```go
// 统一 Config 入口，后端一行切换
cfg := taskgate.Config{
    Broker:  taskgate.FileBroker("./taskgate_data/tasks.db"), // 或 RedisBroker(addr, pw, db)
    Queues: map[string]taskgate.QueueConfig{
        "scoring":   {Workers: 8, RPS: 10, Burst: 20},
        "summarize": {Workers: 2, RPS: 1, LeaseTTL: 10 * time.Minute}, // 长任务给长租约（v5）
        "xunfei":    {Workers: 2},            // 共享网关限额的队列；RPS 不填=0=不限速，只受并发槽约束
    },
    // Type→Queue 路由（v5，问题①）：生产者和消费者共享同一份 Config，
    // 纯生产者进程不注册 handler 也能把任务入对队列。没写的 Type 默认 队列名=Type
    Routes: map[string]string{"review": "xunfei"},
    DefaultQueue: taskgate.QueueConfig{Workers: 4}, // 未配置的队列用这个
}
tg, err := taskgate.New(cfg)

// 注册 handler（按 Type）；进哪条队列由 Config.Routes 决定，Handle 只管执行
tg.Handle("scoring", func(ctx context.Context, t *taskgate.Task) ([]byte, error) { ... })
tg.Handle("review", func(ctx context.Context, t *taskgate.Task) ([]byte, error) { ... })

go tg.Run(ctx)   // 只当生产者的进程不调 Run 即可

// 提交与追踪
id, _  := tg.Submit(ctx, "scoring", payload, taskgate.MaxRetry(3))
task, _ := tg.Get(ctx, id)                 // Status / Result / LastError 都在里面
_ = tg.Cancel(ctx, id)
tasks, _ := tg.List(ctx, taskgate.Filter{Type: "summarize", Status: taskgate.Pending})
stats, _ := tg.Stats(ctx, "scoring")
// 单队列实时水位: {Workers: 8, Running: 3, QueueLen: 120, RPS: 10}
ov, _ := tg.Overview(ctx)
// 全局聚合: map[Type]map[Status]int64，即每种类型各状态的任务数
// 有哪些类型 = key 列表；成功率/占比等比例是纯算术，调用方自己除，库不存比例
res, _ := tg.Wait(ctx, id)                 // 可选的阻塞等结果（内部轮询 Get）

// 优雅停止（v4 审查点 5 + v5 修订）：先停止出队和限流放行 → 等在跑的任务干完 →
// ctx 超时还没干完就 cancel 它们的 context，handler 退出后调度器调 Requeue 把任务
// 放回 pending——不占 Attempts 也不占 LeaseLost，部署重启不该消耗任何配额，
// 否则赶上长任务积压时连续部署 3 次就把健康任务打成 failed 了。
// 进程在 Requeue 前就死掉的极端情况才走租约过期回收（这种算 LeaseLost，罕见可接受）
shutCtx, _ := context.WithTimeout(context.Background(), 30*time.Second)
_ = tg.Shutdown(shutCtx)
```

### 配置注入：库只收 struct，不读环境变量和配置文件（v5 补充）

taskgate 是库不是服务，**自己不读 env、不读配置文件**——从哪读是引用方的事。库内建"读 TASKGATE_XXX"或"找 taskgate.yaml"会跟宿主应用的配置体系（viper/nacos/configmap/纯 env）打架，还引入全局状态，单测想换配置都得动环境变量。

支持方式是把 Config 拆两半：

- **可序列化部分**（Queues/Routes/DefaultQueue 等数值和字符串）：字段带 `json`/`yaml` tag，应用在自己的配置文件里划一段直接 unmarshal 进来
- **只能代码注入部分**（Broker 实例、OnStateChange 回调）：打 `yaml:"-"`，unmarshal 完在代码里补。Broker 的连接参数（Redis 地址/密码、db 文件路径）本来就是构造函数入参，应用从自己的 env/配置里取了传进去

```go
var appCfg struct {
    Redis    RedisConf       `yaml:"redis"`
    Taskgate taskgate.Config `yaml:"taskgate"`   // 应用配置文件里划一段
}
yaml.Unmarshal(b, &appCfg)
appCfg.Taskgate.Broker = taskgate.RedisBroker(os.Getenv("REDIS_ADDR"), pw, 0)
tg, err := taskgate.New(appCfg.Taskgate)         // 校验仍在 New 里 fail fast
```

两个配套细节：

1. `LeaseTTL` 是 time.Duration，标准 unmarshal 不认 `"10m"` 写法，taskgate 提供带 `UnmarshalText` 的 Duration 类型，配置文件里才能写人话
2. 生产者/消费者分进程部署时，两边必须引用**同一份配置**（同一个文件或同一个配置包），Routes/Queues 才一致——New 校验不了跨进程一致性，这条写进文档

### Broker 接口（可插拔的关键）

```go
type Broker interface {
    Enqueue(ctx, task) error
    Dequeue(ctx, queues []string) (*Task, error)       // 阻塞式认领，认领即加租约，返回的 Task 带本次租约令牌
    Ack(ctx, id, leaseToken, result) error             // 成功；同一事务里唤醒依赖它的子任务
    Fail(ctx, id, leaseToken, errMsg, retryAt) error   // 失败 → retrying 或 failed；终态时按策略处理子任务
    Cancel(ctx, id) error                              // blocked/pending/retrying 直接改状态；running 由调度器 cancel context
    FinishCanceled(ctx, id, leaseToken) error          // running 任务被取消、handler 退出后落库成 canceled（v5：Ack/Fail 都表达不了这个终态，接口上原来缺一块）
    Requeue(ctx, id, leaseToken) error                 // Shutdown 超时时归还任务回 pending，不占任何计数（v5）
    Heartbeat(ctx, id, leaseToken) error               // 续租约（M1 由调度器自动调，见第 6 节）
    Get(ctx, id) (*Task, error)
    List(ctx, Filter) ([]*Task, error)
    QueueLen(ctx, queue) (int, error)                  // 供 Stats 用
    Counts(ctx) (map[string]map[Status]int64, error)   // Type×Status 计数矩阵，供 Overview 用
    ReapExpired(ctx) (int, error)                      // 回收租约过期的僵尸任务
    Close() error
}
```

**租约（lease）机制**：worker 认领任务时写入过期时间，进程崩了没 Ack，后台 reaper 把过期任务捞回 pending——"至少执行一次"的保证，两个后端都必须实现（参考方案缺这块，见 13 节）。

**租约令牌防过期竞态（v5）**：每次认领生成一个新令牌（ulid 即可）。经典竞态：worker A 卡死 → 租约过期 → reaper 回收 → worker B 认领开跑 → A 缓过来调 Ack——如果不校验，A 的旧结果会覆盖 B 的执行状态。所以 Ack/Fail/FinishCanceled/Requeue/Heartbeat 全部带令牌，令牌对不上直接拒绝（返回 ErrLeaseLost），A 的调度器收到后丢弃结果就完了。

**接口纪律**：

1. 上层只认 Broker 接口，换后端 = 换构造函数一行（`sqlitebroker.Open` / `redisbroker.New` / `memorybroker.New` 都返回 Broker）
2. 接口是最小公倍数：只收"所有后端都能以相同语义实现"的方法。时间窗口统计这类只有 sqlite 做得动的，不进接口，算后端私有能力
3. **`brokertest` 合规测试包**：把行为契约（认领互斥、阻塞语义、幂等 ID 返回 ErrTaskExists、唤醒原子性、崩溃回收、**Counts 计数一致性**、**过期令牌的 Ack/Fail 被拒**、**retrying 到点后可被重新认领**）写成通用测试函数，三个后端跑同一套题，防止"签名一样、行为漂移"。memory 实现兼任接口语义的参考实现，Counts 等统计一个都不能偷懒（v4，审查附加建议；v5 补令牌与 retrying 两条契约）
4. **`New(cfg)` 时校验配置，fail fast**（v4，审查附加建议）：Workers ≥ 1、RPS ≥ 0（**0 表示不限速，只受 Workers 并发槽约束**，v5 写死语义）、LeaseTTL > 0（不填用默认 60s）、Routes 指向的队列必须已配置或有 DefaultQueue，非法配置返回 error 而不是运行时 panic

## 6. 限流器实现

每队列独立一套 `{Workers, RPS, Burst}`：

- **Workers（并发槽）**：限"同时在跑的任务数"。单机用带缓冲 channel 当信号量；Redis 模式用 Redis 计数信号量（参考 [asynq x/rate](https://pkg.go.dev/github.com/plenigo/asynq/x/rate) 的做法）
- **RPS（启动频率）**：限"每秒新启动多少个"。单机用 `golang.org/x/time/rate` 令牌桶；Redis 模式用 [go-redis/redis_rate](https://github.com/go-redis/redis_rate)（GCRA 算法）
- 两者独立生效，语义写死在文档里，避免使用方误解

**LLM 特化**：handler 返回 `taskgate.ErrThrottled{RetryAfter: 3s}`（对应上游 429/busy）时，**不消耗重试次数**，只按 RetryAfter 延后重排——网关抽风不该把任务打成死信。但不能完全不设限（v5）：网关**永久性** 429（key 配错、账号欠费）会让任务无限重排永不死。所以重排次数记进 `Throttled` 字段，超过上限（默认 100，可配）也进 failed，LastError 记 `"throttled N times"`——上限给得很宽，只拦"永远好不了"的情况，不误伤正常的网关抖动。另有 `ErrSkipRetry` 直接进 failed（参数错误这种重试也没用的）。

普通失败走指数退避 + 抖动：`min(2^n × 1s, 10min) ± 20%`，超过 MaxRetry → failed，保留在库里可查可手动重放。

**租约过期不算业务失败（v4，审查点 1）**：worker 崩了任务被 reaper 回收，这是基础设施故障，不该消耗 MaxRetry——但也不能白回收，否则一个总把 worker 弄崩的"毒任务"会被无限重新调度堵死队列。所以独立计数 `LeaseLost`，超过上限（默认 3，可配）→ failed，LastError 记 `"lease expired N times"`。三种计数各管一段：Attempts 管业务失败、LeaseLost 管 worker 崩溃、ErrThrottled 两个都不占。
（审查建议是直接计入 Attempts + MaxRetry 封顶，更简单；这里有意分开，代价是多一个 int 字段，换来"业务重试配额不被机器故障吃掉"——LLM 任务 MaxRetry 通常只给 2~3 次，被一次 OOM 吃掉一次太亏。）

三种计数的分工（v5 汇总）：Attempts 管业务失败、LeaseLost 管 worker 崩溃、Throttled 管网关限流，各自封顶互不占用；Shutdown 的 Requeue 一个都不占。

**租约 TTL 必须可配，自动续租 M1 就做（v5，问题③）**：`QueueConfig` 加 `LeaseTTL`（默认 60s）。LLM 总结类任务跑几分钟很正常，如果续租排到 M3，M1/M2 阶段健康的慢任务会被 reaper 误回收 → 重复执行 → LeaseLost 三次封顶后被误杀成 failed——精心设计的毒任务防护反过来杀正常任务。所以调度器在任务运行期间每 LeaseTTL/3 自动调一次 Heartbeat（对 handler 完全透明），这个 M1 就得有；M3 才做的是暴露给 handler 的手动续租口子（handler 想自己控制租约节奏的场景）。

## 7. 两个后端怎么实现

### 7.1 文件后端：单文件 SQLite（modernc.org/sqlite，纯 Go 免 cgo）

- 一张 `tasks` 表 + `(queue, status, run_at)` 索引；认领用一条事务内的 `UPDATE tasks SET status='running', lease_token=?, lease_until=? WHERE id = (SELECT id FROM tasks WHERE queue IN (...) AND status IN ('pending','retrying') AND run_at <= now ORDER BY run_at, created_at LIMIT 1) RETURNING *`，事务内更新天然防重复认领。两处 v5 修订：**子查询写法**不依赖 SQLite 的 `UPDATE ... ORDER BY ... LIMIT` 编译开关（那是可选特性，纯 Go 驱动开没开要赌）；**认领条件包含 retrying**——原方案只查 pending，Retrying 状态的任务到点后永远没人捞（Redis 后端不存在这个问题，delayed zset 搬回 pending list 时状态就翻回 pending 了；两个后端对外语义一致：retrying 到点即可被认领，brokertest 测这条）
- WAL 模式，同机多进程也能安全共用一个 db 文件
- 出队阻塞语义（v4，审查点 4）：**同进程内** Submit 直接条件变量唤醒，零延迟；**跨进程**共用 db 文件时没有通知通道，只能靠 100ms 轮询兜底，最坏 100ms 延迟，文档写明。Dequeue 的合同统一定义为"阻塞到有任务或 ctx 取消，实现内部允许轮询"，brokertest 只测语义不测手段
- 对使用方来说它就是"一个本地文件"，符合"文件后端"的预期，但省掉了自己造原子写、索引、崩溃恢复的活（对比参考方案的 JSON 目录方案，见 13 节）
- `Counts` 就是一句 `GROUP BY type, status`，统计白送；时间窗口类统计（今天的失败率等）也能靠 WHERE 直接查

### 7.2 Redis 后端

- `tg:pending:{queue}`：每队列一个 list
- 阻塞出队（v4 审查点 4 + v5 修订）：原方案"BLMOVE 搬进 in-flight zset"是写不出来的——**BLMOVE 只能 list→list，进不了 zset**。改成两步：`BLMOVE tg:pending:{q} → tg:claiming:{q}`（每队列一个中转 list，带短超时循环以响应 ctx 取消）先原子拿到任务，再跑 Lua 把它从中转 list 挪进 in-flight zset 并写租约/状态/令牌。两步之间进程崩了：任务滞留在中转 list 里，reaper 除了扫 zset 的过期项，**也要扫各 `tg:claiming:{q}` 里滞留超时的任务**搬回 pending——崩溃窗口仍然被 reaper 覆盖，只是扫描范围多一处，brokertest 的崩溃回收用例要盖住这个窗口
- `tg:inflight`：zset，score=租约到期时间，reaper 定期把过期的移回 pending
- `tg:delayed`：zset，score=RunAt，到期搬回 pending（延迟执行和重试退避都走这里）
- `tg:task:{id}`：hash 存任务详情；另按状态维护索引 set 供 List/Stats 查询
- `tg:stats`：hash，字段 `{type}:{status}`，**每次状态流转的 Lua 里顺手 HINCRBY**（旧状态-1 新状态+1）。所有流转本来就必须走 Lua（唤醒原子性要求），所以计数器不会漂；`Counts` 读一次 HGETALL 即 O(1)。`tg:types` set 记录出现过的类型
- 运维可直接 `redis-cli LLEN tg:pending:scoring` 看各类型积压
- 诚实的限制：redis 后端只有"累计计数 + 当前水位"，时间窗口类统计（今天的成功率）不做——要这个用 sqlite 后端，或把任务终态事件接到你自己的监控里（M3 的 OnStateChange 钩子）

### 7.3 memory 后端

纯内存不落盘，跑单测和 mock 用，顺手送一个。

## 8. 取消（Cancel）语义

- `blocked`（v4，审查点 3）：直接改成 canceled，不用等父任务；**并触发依赖传播**——它自己的子任务按 FailFast 策略连锁取消（IgnoreParentFailure 的子任务照常算依赖满足）
- `pending/retrying`：直接改成 canceled，不再被认领
- `running`：调度器为每个在跑任务保存其 cancel func，收到取消请求就 cancel context，handler 自己决定多快退出；退出后调度器调 `FinishCanceled(id, leaseToken)` 落库成 canceled 并触发依赖传播（v5：这条落库路径原来在接口上是缺的——Ack 是成功、Fail 是失败，都表达不了"被取消"）
- Redis 多实例下，running 任务可能在别的进程上——v1 先只保证"本进程在跑的能即时取消，别的进程等下一次 Heartbeat 时发现取消标记再退出"，简单可靠

## 9. 任务依赖（v3 新增，克制版）

**原则：只做"一级依赖（父任务列表）+ 就绪状态机"，不做 DAG 编排引擎。** 链式流水线（检索→生成→打分）靠"每个任务都能声明父任务"自然串起来，多个父任务就是 fan-in，不需要静态图定义。

### 9.1 模型

- Submit 时带 `taskgate.DependsOn(ids...)`；只要有未结束的父任务，初始状态就是 `Blocked`，不进队列
- 父任务全部结束后被唤醒：`Blocked → Pending` 入队，之后**照常受所属队列限流**——依赖不能绕过闸门
- 父任务失败/取消时子任务怎么办，提交时配置：
  - **`FailFast`（默认）**：子任务直接 canceled，LastError 记 `"parent <id> failed"`，并继续向下传播（孙任务也连锁取消）
  - **`IgnoreParentFailure`**：父任务只要结束（无论成败）就算依赖满足，照常唤醒

### 9.2 唤醒机制：事件触发，不轮询

调度器不去轮询所有 Blocked 任务查依赖（O(N²) 白烧）。任务 A 到达终态（Ack/Fail/Cancel）时，在**同一个事务 / 同一段 Lua 脚本里**一口气完成：

> 改 A 的状态 → 查反向索引找到依赖 A 的子任务 → 各子任务"未完成父数"减一 → 减到 0 的翻成 pending 入队

原子性是生命线：如果分两步写，崩在中间就是"丢唤醒"，子任务永远卡在 Blocked。

存储层适配：

- **sqlite**：加一张 `task_deps(child_id, parent_id, done)` 表 + 子任务行上的 `pending_parents` 计数，一个事务全搞定
- **redis**：`tg:children:{parentID}` set 做反向索引 + `tg:waitcnt:{childID}` 计数器，Lua 原子递减，减到 0 就 LPUSH 进 pending
- **兜底**：reaper 顺带扫"Blocked 但父任务其实都已结束"的任务做防御性修复，正常路径不靠它

**连锁取消用链式触发，不做一次性递归（v4，审查点 2）**：FailFast 传播如果在一个事务/Lua 里递归取消整棵子树，子孙多了就是长事务。改成统一规则：**任何任务到达终态（completed/failed/canceled）都触发同一段"处理直接子任务"的逻辑，只管一层**——failed/canceled 时按策略取消直接子任务，子任务变 canceled 本身又是一次终态事件，再触发它自己那层，逐层往下传。每步都是小事务，最终一致；传播链路上崩了，reaper 的防御性扫描兜底补齐。

### 9.3 天然无环，不需要环检测

提交时校验：**所有父任务 ID 必须已存在，否则拒收**。依赖边只能指向"更早提交的任务"，图上根本画不出环——比在 Lua 里做环检测简单得多也可靠得多。代价是不能"先提交子、再提交父"，这个限制写进文档就行。

竞态处理：校验和落库在同一事务里；如果提交那一刻父任务已经全部完成，直接以 pending 入队，不会卡死。

两个边界（v5）：

- 提交那一刻父任务已经是 **failed/canceled** 且策略是 FailFast → 子任务直接以 canceled 落库（LastError 记 `"parent <id> failed"`），不入队——和"父任务先成功后失败"走同一套规则，不留"已死的父唤不醒的子"这种悬空态
- 结果清理交给使用方（见第 10 节），但**还被子任务引用着的父任务不能清**：父任务记录删掉后，新提交的子任务会被"父必须存在"校验误拒，reaper 的防御性扫描也分不清"父任务完成后被清了"和"父任务从来不存在"。这条约束写进文档，清理示例里给出"先查 `tg:children:{id}` / task_deps 无引用再删"的写法

### 9.4 父任务的结果怎么传给子任务

不做自动结果注入（克制）。子任务 handler 里拿 `t.DependsOn` 自己 `tg.Get(ctx, parentID)` 读父任务的 Result——显式、好调试。需要更复杂的动态编排（条件分支、父任务跑完再决定生成什么子任务），用户在 handler 里自己 Submit 就是了，**用组合代替配置**。

### 9.5 用法

```go
sumID, _   := tg.Submit(ctx, "summarize", payload)
scoreID, _ := tg.Submit(ctx, "scoring", scorePayload,
    taskgate.DependsOn(sumID))          // 提交即 Blocked

t, _ := tg.Get(ctx, scoreID)
// t.Status == Blocked，t.DependsOn 里能看到在等谁
```

## 10. 明确不做的（YAGNI）

- ❌ Web UI / 管理后台（状态用 `List`/`Stats` 自己包个接口）
- ❌ cron 周期调度（一次性延迟任务库原生支持：`Delay/RunAt` 选项走 run_at/delayed zset）。周期任务的配方：应用里 robfig/cron 触发 Submit + **确定性任务 ID**（如 `daily-report:2026-07-14`）做幂等去重——多实例同时触发也只入队一份，不需要选主和分布式锁。前提：Enqueue 遇到已存在的 ID 返回 ErrTaskExists，这个语义要实现
- ❌ DAG 工作流引擎：静态图定义、条件分支、循环、自动结果注入都不做——一级依赖 + handler 内自由编排已覆盖 LLM 流水线场景（见第 9 节）
- ❌ 独立部署的 server 模式（就是个库）
- ❌ 内建环境变量/配置文件读取（库只收 Config struct，字段带 yaml/json tag 方便应用自己 unmarshal，见第 5 节"配置注入"）
- ❌ 结果长期存储/TTL 管理（v1 结果存任务记录里，清理交给使用方；v5 注意：还被子任务引用的父任务不能清，见 9.3）
- ⏸ 优先级调度、状态变更 webhook 回调：参考方案里有，先不做。但 `OnStateChange func(Task)` 的**注册 API M1 就暴露**（默认空实现，同进程内回调，方便埋点监控）；webhook 等重语义 M3 再说（v4，审查附加建议）

## 13. 与参考方案的差异（审查重点）

采纳了的：Type 一等公民 + 类型级限流、Cancel、Stats/List、统一 Config、`taskgate` 命名、竞品补充（taskq/pond 等）；v3 又采纳了：一级依赖 + Blocked 状态 + 事件唤醒 + FailFast/IgnoreParentFailure 策略 + 反向索引存储。以下是**有意没照抄**的几处，请审查时确认：

1. **文件后端用 SQLite，不用"JSON 目录 + pending.txt"**。参考方案的目录方案要自己处理：原子写（临时文件+rename）、pending.txt 与任务文件的一致性（两处写，崩在中间就对不上）、按状态/时间查询（只能全目录扫描）。SQLite 一张表全解决，纯 Go 驱动零依赖，对外仍是"一个本地文件"
2. **Payload/Result 用 `json.RawMessage`，不用 `interface{}`**。任务要落盘/进 Redis，`interface{}` 序列化再读回来类型就丢了（map[string]interface{} 满天飞），还会诱导使用方存不可序列化的东西。收 `[]byte`/结构体自己 marshal，边界清晰
3. **补了租约 + reaper**。参考方案里 worker 崩溃后 running 任务就永远卡住了，没有回收机制。这是队列库的生命线，必须有
4. **保留 `Retrying` 状态和 `ErrThrottled` 特化**。参考方案失败即 failed；但 LLM 场景下"网关 busy 等重试"和"真失败"必须区分开，否则状态查询没法看
5. **环检测改成"父任务必须已存在"的提交校验**（v3）。参考方案打算在 Redis 后端用 Lua 做环检测；其实只要依赖边只能指向已提交的任务，环根本画不出来——一条校验规则替代一套图算法。代价是不能"先提交子、再提交父"
6. **不做父结果自动注入子 payload**（v3）。子任务 handler 自己 `Get(parentID)` 取，显式好调试；自动注入要定协议、要处理多父合并，复杂度不值
7. **FailFast 的取消向下传播**（v3）。参考方案只处理了一层；父失败→子取消后，孙任务也必须连锁处理，否则孙永远 Blocked

## 14. 风险与对策

- **SQLite 并发写锁**：WAL + 写事务毫秒级，单机到不了瓶颈；真到了就是该切 Redis 的信号
- **Redis Lua 脚本正确性**：M2 用 miniredis 写集成测试 + 真 Redis 跑竞争测试
- **多进程 Cancel 时效**：v1 接受"跨进程取消有 Heartbeat 间隔的延迟"，文档写明
- **丢唤醒（依赖功能最大的坑）**：终态更新与子任务唤醒必须同事务/同 Lua；另加 reaper 防御性扫描兜底；M1 专门写"唤醒中途 kill 再恢复"的测试
