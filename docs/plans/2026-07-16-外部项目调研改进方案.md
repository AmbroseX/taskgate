## 1. 结论先行

**一项可立即做,两项退回 research,三项挂起:**

1. **【✅ 批准立即做|文档缺陷】README 声称"恰好执行一次"是错的,且全库没有 handler 幂等契约**(第 7 节)。taskgate 实际是 at-least-once,README 自己在别处写了"崩溃后捞回重跑",却在 Redis 后端一节里同一句话内既说"恰好被执行一次"又说"回收重跑"。目标场景是花钱的 LLM 网关,用户照这话不做幂等防护会被重复扣费。**纯文档改动,不动代码。已按评审 #1/#2/#11 修正后实施完毕**(删掉 `OnStateChange` 副作用配方、把"本地副作用"与"外部调用"分层讲、收紧"认领互斥"措辞)。
2. **【⛔ 退回 research|需求成立,设计不成立】终态任务的 ID 永久占死**(第 6 节)。**需求确凿**:README 的 cron 配方(确定性 ID + `ErrTaskExists`)在任务失败后无法重提,同 ID 拒绝无条件、且全库没有删除 API。**但 v1 的设计有阻断级问题**:`Delete` + ID 复用会静默篡改历史依赖的指向(评审 #3),而且原子性/索引清理/并发竞态被严重低估(评审 #4)。**任务身份语义必须先裁决,不进 spec**(v3 把"原地重跑 `Retry`/`Replay`"补为正式候选并倾向先评它,见 6.4)。
3. **【⛔ 退回 research|需求成立,接口不成立】周期配额 Periodic Quota**(第 5 节)。**需求确凿**:限流只有频率维度,缺配额维度,而 LLM 网关的限额主要是配额制。**但 v1 的接口不成立**:`CommitQuota()` 无 ctx 无返回值,无法表达失败(评审 #5);`INCR`+`EXPIRE` 分两步有"永久锁死队列"的崩溃窗口;**且"退回进程内配额"会让这个功能在多进程下直接失效**(评审 #6)。**必须先裁决硬配额 / 尽力配额,再定接口。**
4. 三项挂起(第 9 节):熔断、队列暂停/恢复、消息过期丢弃。都能说清价值,但都没有真实诉求方,按项目一贯的 YAGNI 纪律等需求出现。

**明确不做的**(第 8 节):funboost 侧两项(`Wait()` 改事件驱动 8.1、`Task.Metadata` 8.2);dask 侧三项(LIFO 队列 8.3、静态优先级 8.4、5 钩子回调 8.5),另 dask 的 `annotate` 并入 8.2 一起讨论。第 4 节列出"已核实不是缺口"的项,防止后续审查重复讨论。

**两个参考项目的定位差异必须先讲清,功能清单都不能照抄:**

- **funboost** 是 791 个 py 文件的全能平台,自称 40+ 消息队列、30+ 控制功能,还塞了爬虫、FaaS、CDC、Web 管理台。
- **dask** 是 DAG compute engine,任务是进程内的 `func + args`(`_task_spec.py:636` 的 `Task` 直接持有 `func: Callable`),目标函数是**最小化内存驻留**。它的大部分设计(静态排序、LIFO、重算换内存、图剪枝)都是为这个目标服务的,而 **taskgate 的结果存在 broker 磁盘/Redis 里、是花钱调 LLM 换来的、绝不能重算**——目标函数正好相反。
- taskgate 是 12k 行 Go 的聚焦库,README 第一句写明"是库不是服务"。方案 v5 第 13 节和 M3 完成记录里的"维持不做"清单已经把优先级调度、Web UI、DAG 引擎、webhook 一一否决过。

**dask 的最大价值不在功能,在两处**:① 它的任务契约文档(`docs/source/graphs.rst:111-143` "Task Expectations")给了第 7 节那个文档缺陷的标准修法;② 它的中间结果引用计数(`local.py:290-317`)改进了第 6 节 `Delete` 的约束设计。另有三处**印证了 taskgate 既有裁决是对的**(4.8~4.10),不用改但值得记档。

## 2. 调研的可信度说明

funboost 的 README 语气极度自夸("吊打"、"完虐 celery"、发布性能 22x / 消费性能 46x),这些**不作为本方案的依据**。

但它的**源码注释质量和取舍诚实度远高于营销文案**,本方案的判断主要基于源码。举例:它会主动写"`function_timeout` 会降性能且可能死锁(函数拿着锁被杀),非必要别用"、"`wait_for_possible_has_finish_all_tasks` 是不精确判断,慎用"、"kombu 性能差 5-10 倍,除非我们没实现的中间件否则别用"。这类注释才是有价值的部分。

另外它所谓 40+ broker 中有相当比例是套壳(kombu/celery/rq/dramatiq/huey),真正独立实现的核心是 RabbitMQ 系 + Redis 系 + Kafka + 内存队列。

## 3. taskgate 明显更强、不要被宣传语误导的地方

这一节是为了防止审查时按 funboost 的功能清单反向要求 taskgate。

| 维度 | funboost 的做法 | taskgate 的做法 | 判断 |
|---|---|---|---|
| **分布式限流** | `qps / 活跃消费者数量` 本地平分(`base_consumer.py:613-617`),靠 Redis 心跳统计消费者个数(10s 心跳 / 25s 过期) | redis_rate 的 GCRA 全局令牌桶 | **taskgate 更硬**。funboost 方案热路径零 IO、性能好,但消费者数量变化时有 10~25s 窗口偏差,且假设每个消费者拿到的消息量均等。作为"廉价近似"有价值,但不是精确限流 |
| **杀死运行中任务** | ctypes 强杀线程(`kill_remote_task.py`),它自己文档承认"可能造成死锁" | handler ctx cancel | **taskgate 更干净**,Go 天然优势 |
| **租约/孤儿任务回收** | 心跳 + per-consumer unack zset + 独立 registry 索引 + 分布式锁下的回收任务 | 每任务一个租约,心跳自动续,reaper 回收 | **等价,taskgate 更简洁**(详见 4.1) |

## 4. 已核实"不是缺口"的项(防止重复讨论)

调研中有几项被列为"funboost 的高价值设计",但核对 taskgate 源码后确认**已经解决、或 taskgate 的解法更好**。记录在此,避免后续审查再提。

### 4.1 心跳级 ACK —— 不是缺口

funboost 批评的是 Celery 的 `visibility_timeout` 方案:纯超时判定分不清"执行慢"和"进程死",长耗时任务会被反复误判重回队列。它的解法是按进程存活判定(心跳 set + per-consumer unack zset)。

**taskgate 没犯这个错**:租约是**被心跳续的**(`scheduler.go`,每 `LeaseTTL/3` 一次)。慢而活着的 worker 一直在续租,reaper 只在心跳停了(进程死了)才回收。所以 taskgate 的死亡判定**本质上就是心跳判定**,只是表达成"每任务一个租约"而不是"每消费者一个 unack 集合 + 成员表",反而更简洁——不需要 funboost 那个额外的 `funboost_unack_registry` 索引补丁(它自己注释里写明那是踩坑后加的"方案C")。

唯一的例外是 `ManualHeartbeat=true` 手动档,租约可能在 worker 活着时过期。但那是 opt-in,且 godoc 与 README 已写明"handler 对自己的租约负全责"。

差异只在检测延迟。**初稿写的"taskgate ≤ `LeaseTTL`"是错的(评审 #10)**,漏算了 reaper 的扫描周期——`scheduler.go:185` 是 `s.reapLoop(ctx, minTTL/2)`,即每 `min(各队列 LeaseTTL)/2` 扫一次。准确的分解应当是三段:

| 段 | 时长 | 说明 |
|---|---|---|
| 租约失效 | `LeaseTTL` | 从最后一次成功续租算起 |
| reaper 发现 | 0 ~ `LeaseTTL/2` | 取决于崩溃点落在扫描周期的哪里 |
| 重新被认领 | 取决于队列水位与限流 | 回到 pending 后要重新排队、占槽、等令牌 |

所以**从最后一次成功续租到任务被 reaper 捞回,单队列下是 `LeaseTTL` ~ `1.5×LeaseTTL`**(默认 60s → 60~90s),再加后端与调度延迟才是"重新开始跑"。对比 funboost 的 ≤25s(10s 心跳 / 25s 过期,固定)。

两者都可接受,但**这个数字要如实写**——它直接决定用户怎么配 `LeaseTTL`。README 的"跨进程延迟(心里有数)"一节应当同步核对是否有同样的漏算。

### 4.2 重试的 sleep 模式 vs requeue 模式 —— 不是缺口

funboost 的价值主张是"长退避间隔下 sleep 会占着 worker 拖垮吞吐,应该用 requeue + countdown 把消息扔回队列、立即释放线程"(`base_consumer.py:840`)。它默认是 sleep 模式,requeue 要显式开。

**taskgate 从一开始就只做 requeue 那种**:失败 → `retryAt = now + backoff(Attempts)` → 回 `retrying` 状态 → worker 立刻释放。没有 sleep 模式,也就没有选择困难。**已经是好的那个。**

### 4.3 定时任务的分布式锁加在 fetch 阶段 —— 不适用,且 taskgate 的解法更好

funboost 把 Redis 分布式锁加在 APScheduler 的 `_process_jobs`(扫描并取出到期 job)而不是用户函数上,取出即从 jobstore 删除,其他实例取不到,从根本上防重复执行(`timing_job/apscheduler_use_redis_store.py`)。这个设计确实比"给 job 执行加锁"优雅。

**但 taskgate 根本不做 cron**(方案 v5 第 10 节明确非目标)。README 给的配方是"应用侧 robfig/cron 触发 Submit + 确定性任务 ID 靠 `ErrTaskExists` 幂等去重"——**不用分布式锁就实现了同一周期键的提交去重**,没有惊群、没有锁续期、没有锁超时误放。在"防止多实例重复提交"这件事上,taskgate 的答案更简单。

> **⚠️ 措辞修正(评审 #9)**:初稿这里写的是"不用锁就达成**恰好一次**",**这个说法是错的,和本方案第 7 节的结论直接冲突**。确定性 ID + `ErrTaskExists` 保证的是**同一周期键最多创建一条任务记录**(提交去重),**不保证 handler 只执行一次**——任务入队后照样会被崩溃回收重跑。准确表述:
>
> **无需分布式锁即可实现同一周期键的提交去重;任务执行仍是 at-least-once,handler 必须幂等。**
>
> 这个错误本身正说明第 7 节那个文档缺陷的传染性:"恰好执行一次"这个说法一旦在项目里立住,连写方案的时候都会顺手复用它。

(另外这个配方还有个洞,见第 6 节。)

### 4.4 有界队列做背压 + 控频在投递端 —— 不是缺口

funboost 的 `FlexibleThreadPool` 用 `work_queue_maxsize=10` 的有界队列做背压,并把 QPS 控制放在"往并发池投递"而非"函数执行",这是它能做到 0.00001 ~ 50000 QPS 的原因。

**taskgate 的 `claimLoop` 本来就是投递端控制**:先占槽 → 再等令牌 → 最后 Dequeue(`scheduler.go:252-263`,注释写明了"先拿令牌再等槽会白烧令牌"这个取舍)。`Workers` 天然是并发上界。已经对了。

### 4.5 `fct` 隐式任务上下文 —— taskgate 更好

funboost 用 `contextvars` 做隐式上下文,函数内任意深度可拿 `fct.task_id`,还得配一个 `FctContextThread` 解决"新开线程丢失 contextvars"的问题。

taskgate 直接把 `*Task` 传给 handler,显式、无坑。

### 4.6 `broker_exclusive_config` 白名单机制 —— Go 里不需要

funboost 每种 broker 注册自己支持的独有配置 + 默认值,用户传了不支持的 key 会告警。这是 Python 弱类型下的补救。

taskgate 的 `redisbroker.Options{Addr, Password, DB, KeyPrefix}` 这种每后端独立的结构体,**编译期就把这事解决了**。

### 4.7 Broker 接口反转(`Run(submit func(*Task))` 取代 `Dequeue()`)—— 不建议

调研把这个列为最高价值借鉴点。funboost 的 `_dispatch_task` 哲学是"我不管你怎么拿到任务,你拿到后调 `_submit_task(msg)` 就行"(`base_consumer.py:470-496`),这是它能把 MySQL binlog CDC、watchdog 文件事件、TCP socket、甚至整个 Celery 塞进同一个 broker 抽象的原因。

**对 taskgate 不成立**:
- Broker 接口已冻结,五后端过 18 条契约
- `docs/plans/2026-07-15-MySQL-PG后端适配方案.md` 第一句就是"Broker 接口一行不用改",整个方案建立在现接口上
- taskgate 不追求支持 40 种 broker,`Dequeue()` 假设的 pull 模型对 memory/sqlite/redis/mysql/pg 全部成立

为了一个不想要的自由度掀掉地基,代价和收益不成比例。**funboost 需要这个是因为它要当平台,taskgate 是库。**

### 4.8 dask 印证:"不做父结果自动注入子 payload"(方案 v5 第 13 节第 6 条)是对的

dask 的 `docs/source/spec.rst:168-186` 有一节 **"Why not tuples?"**,讲他们为什么放弃 `(add, 'x', 'y')` 这种紧凑表示、换成 `Task("z", add, x.ref(), y.ref())` 的显式类型。三条理由:

1. **self-describing vs context-dependent**:`(func, "x", "y")` 的含义取决于它所在的图——`"x"` 到底是字面量还是对其他任务的引用,**看这个 tuple 本身根本不知道**
2. **性能**:要靠"递归遍历 + 与全局 key 表比对"才能解释一个 task,大图上是瓶颈
3. **正确性**(最狠的一条,原文):

> *"For APIs that allow users to define their own key names this can further cause **false positives where intended literals are replaced by pre-computed task results**."*

即:用户自定义 key 名时,**本该是字面量的字符串被当成引用替换掉了**——一个 silent 的数据损坏 bug。

**taskgate 根本没创造这个歧义**:方案 v5 第 13 节第 6 条决定"不做父结果自动注入子 payload,子任务 handler 自己 `Get(parentID)` 取,显式好调试"。一旦做了自动注入,payload 里就必然出现"这个字段是字面量还是上游结果引用?"的问题,然后就要定协议(`$ref` 前缀之类)、要处理多父合并——正是 dask 花了一个大版本去掉的东西。

**dask 现在的解法是在类型层面显式区分**:`DataNode`(字面量)、`TaskRef`(引用)、`Task`(计算)、`Alias`(重定向)。如果 taskgate 将来真要做注入,这是唯一正确的路子——**用 tagged union,绝不用命名约定**。但更好的选择是继续不做。

**记档意义**:第 13 节第 6 条当时的理由是"自动注入要定协议、要处理多父合并,复杂度不值",属于成本判断。dask 的经历补上了一条更硬的理由:**这个抽象本身会产生 silent 数据损坏**。

### 4.9 dask 印证:"父任务必须已存在"消灭环检测(方案 v5 第 13 节第 5 条)是对的

dask 必须实现真正的环检测:`dask/core.py:362` 的 `_toposort(returncycle=True)` + `getcycle`,而且 `order.py:184-190` 还得靠"`ndependencies` 结果长度对不上"来先判断有没有环,有环才花大代价定位——因为它的图是用户任意构造后一次性提交的,环画得出来。

taskgate 的 Enqueue 校验"父任务必须已存在"(`specs/001-m1-core-queue/spec.md:94`),依赖边只能指向已提交的任务,**环根本画不出来**。一条提交校验替代一套图算法,代价只是"不能先提交子、再提交父"。

### 4.10 dask 印证:显式 `WithID` 优于自动指纹

funboost 的去重是按函数入参 md5 自动算指纹(见 6.2)。dask 也走内容哈希路线(`tokenize.py`),但它为此踩的坑值得看——`dask/tokenize.py` 489 行几乎全是在处理确定性哈希的边界情况:

- **set 顺序**(`tokenize.py:136-138`):*"in some Python version / OS combinations, **set order changes every time you recreate the set** (even within the same interpreter)"* → 强制 `sorted(s, key=str)`
- **pickle 字节不稳定**(`tokenize.py:233-263`):**连续 pickle 同一对象最多 3 次,要求连续两次字节完全相同**才接受,否则报错
- **`object()` 无内容可哈希**(`tokenize.py:222-230`):**故意永久泄漏对象**(`_seen_objects` 全局 set 永不清)以防 id 被复用
- **循环引用**、**pyarrow 空 validity map**、**pandas Arrow 数组碎片化**各有一套归一化

最关键的是它有个 `ensure_deterministic` 开关(`tokenize.py:83-89`):**内容无法确定性哈希时直接抛 `TokenizationError`,而不是悄悄退回 `uuid.uuid4()`**。

**taskgate 让用户显式给 ID,上面的坑一个都不用踩。** 这条印证了 `WithID` 的设计选择;若将来真要加"按 payload 自动去重",必须同时提供 dask 那种"算不出确定性指纹就报错"的出口,否则就是 silent 的错误去重。

## 5. 改进项 1(⛔ 退回 research):周期配额 Periodic Quota

> **本节现行立场(v3 收口)**:**需求成立**(5.1),**能力声明与耗尽行为的约束已定**(5.1 收窄声明、5.3 窗口与耗尽语义),**接口与实现位置未定**——v1 的 `WaitQuota`/`CommitQuota` 接口和"退回进程内配额"均已否决,草案存档见附录 A.1。往下推进要先过第 13 节的裁决 #2(硬配额/尽力配额)、#3(非 redis 后端怎么办)、#4(只做固定时长窗口)。

### 5.1 需求论证

funboost 的 `contrib/override_publisher_consumer_cls/periodic_quota_mixin.py` 把这件事讲透了,原话:

> 假设 ChatGPT 允许你每天使用 24 次,不代表你每使用一次然后需要间隔 1 小时才能再次使用。你可以一口气把一天额度快速用完,然后当天不用就好了。**所以周期额度和运行频率是两码事**,周期额度不代表你要把额度次数除以周期时长然后匀速执行。

它顺带指出 Celery 的 `rate_limit='24/d'` 实现成了"每次间隔 1 小时",是错的抽象。

**这正打中 taskgate 的核心场景。** `QueueConfig` 现在只有 `{Workers, RPS, Burst, LeaseTTL, ManualHeartbeat}`,限流全是频率维度。而 LLM 网关的计费和限额**主要是周期配额制**,不是频率制。

> **⚠️ 能力声明必须收窄(评审 #7,P1)**。初稿这里写的是"每天 N 次调用、每月 N token",**两个说法都超出了这个设计能给的东西**:
>
> | 初稿的说法 | 为什么给不了 |
> |---|---|
> | **每月** N 次 | **自然月不是固定 `Duration`**(28/29/30/31 天)。`QuotaPeriod Duration` 表达不了"下个月 1 号重置" |
> | 每 **token** | `QuotaLimit int` 数的是**任务被放行的次数**,不是 token 数。token 用量只有 handler 调完网关才知道,**需要一套"执行后上报实际用量"的模型**,和当前这个"放行前扣减"的模型是两回事 |
> | 每 **天**(自然日) | 见 5.3 的时区问题——对齐 epoch 得到的是 UTC 零点,且 `QuotaOffset Duration` **处理不了夏令时** |
>
> **第一版的能力声明只能是这一句**:
>
> > **在固定时长窗口内,最多启动 N 次 handler。**
>
> 不得描述成自然月控制、token 账单控制。另外还有两个"配额被消耗但没真正调网关"的漏口要写进合同:**handler 在真正调网关之前就失败/被取消,以及任务类型未注册 handler 被判 `FailSkip`,都已经消耗了一次配额。**

taskgate 的定位是"调用 LLM / OCR 这类有配额的外部网关"(README 第一段),却没有配额维度——这是设计上的真空白。

**频率和配额正交,应当共存**:
- `RPS=3` 管"别把网关瞬时打爆"
- `QuotaLimit=5000, QuotaPeriod=24h` 管"**每个固定 24h 窗口最多启动 5000 次 handler**"——注意这是对齐 epoch 的固定时长窗口,**不等于当地自然日,也对不齐月底账单周期**(见 5.3;自然日/账单对齐是第二版的事)。它能给的是"长期日均用量有上界"这一层保护,不是精确的账单控制

### 5.2 配置形态(未定)

配置最终长什么样,取决于 13 节裁决 #2/#3/#5 的结果(尽力配额的超发合同怎么表述、各后端共享计数怎么落、窗口时间来源)。v1 的 `QueueConfig` 字段草案(`QuotaLimit`/`QuotaPeriod` + 校验规则)存档在附录 A.1,裁决后重新起草,不作为现行建议。

### 5.3 语义定义(需人工确认)

**窗口模式:只做固定窗口,不做滑动窗口。**

funboost 支持两种(`sliding_window` 参数):滑动窗口从进程启动时刻起算,固定窗口对齐整点边界。

**建议只做固定窗口**,理由:
- **窗口边界必须全局一致**:固定窗口对齐 epoch,多进程/多机各自算出的窗口起点相同,计数才共享得起来;滑动窗口"从进程启动起算"在多进程下各进程窗口起点不同,配额会超发,语义混乱
- YAGNI:没有真实场景要滑动窗口

(v3 更正:初稿这里还有一条"配额的用途是对齐网关计费周期,网关按自然日/自然月计费,对齐整点才有意义"——**这条理由不成立,删除**。第一版明确对不齐自然日/自然月(见 5.1 的收窄声明和下面的时区段),不能一边收窄能力声明一边拿"对齐账单"当论据。)

固定窗口的边界:`QuotaPeriod` 对齐 Unix epoch 的整数倍(`now / period * period`)。`24h` 周期即对齐 UTC 零点。

**⚠️ 对齐 epoch 只保证"公式相同",不保证"时刻相同"(第三轮评审 #1)。** 多机部署存在时钟偏差:窗口边界附近,节点 A 还在旧窗口、节点 B 已进新窗口,两个窗口 key **同时放量**。所以"窗口时间用谁的钟"是个必须裁决的问题:
- **硬配额**不能依赖应用节点时钟,必须用**共享介质的服务端时间**(redis 的 `TIME`、SQL 的 `now()`)——但这和"所有时间由注入 clock 提供"的测试铁律直接冲突(服务端时间怎么被 fakeclock 控制?redis 侧有 ARGV 注入时间的既有先例可循,SQL 侧没有),这个调和方案是 research 的必答题
- **尽力配额**可以用节点时钟,但"时钟偏差引起的边界双放"要并进超发合同一起表述

**⚠️ 时区:第一版不解决,先不选(评审 #7 修订)。** 网关配额通常按**当地时区**的自然日重置,而对齐 epoch 得到的是 UTC 零点。初稿列了三个选项并倾向 (b) 加 `QuotaOffset Duration`——**(b) 是错的**:固定偏移**处理不了夏令时**,DST 切换那两天窗口会错位一小时,而这恰恰是"对齐账单"最不能错的地方。

**修订后的立场:第一版只支持固定时长窗口(对齐 epoch),文档写明"不是自然日/自然月"。** 先不引入任何时区概念。

若后续确认"必须按自然日重置"是真需求,那时再设计,且**必须用 IANA 时区名**(`Asia/Shanghai` 而不是 `+08:00`)并明确定义 DST 行为——`Duration` 那套文本解码路子届时不适用,Config 的 yaml 化方式要一起想。**这属于第二版的事,不在本方案范围。**

**配额耗尽时的行为:停止认领,不占用 worker 槽。**

配额用完后该队列**停止认领新任务**,等到下一个周期窗口开始再恢复。关键约束:
- **不能占着 worker 槽等**——周期可能是一天,占槽等于永久泄漏并发额度
- **不能把任务认领出来再拒绝**——那要 Requeue,且会污染任务状态
- 已在跑的任务**不受影响**,正常跑完(配额管的是"放行",不是"中断")
- 配额耗尽**不是错误**,不进 `Throttled` 计数、不进 `failed`。任务老实待在 `pending` 等下个周期

这一点和 `ErrThrottled` 有本质区别:`ErrThrottled` 是**网关告诉你它忙**(任务已经发出去了),配额耗尽是**我们自己知道不该发**(任务还没发)。

### 5.4 实现位置(未定;本节保留否决理由与约束)

实现位置没有现行设计,只有已核实的约束。事实基础:现有 `QueueLimiter` 接口(`limiter.go:16-23`)是两个正交维度——`AcquireSlot`/`ReleaseSlot` 限"同时在跑多少",`WaitToken` 限"每秒新启动多少";配额是第三个维度。v1 的 `WaitQuota()/CommitQuota()` 接口草案已否决,存档见附录 A.1。

**⚠️ 周期配额不能直接复用现有 `LimiterProvider`(第二轮评审,P1)。** 这个接口是**整包替换**的:`LimiterProvider` 返回一个同时负责 slot 和 RPS 的完整 `QueueLimiter`(`broker.go:62`),scheduler 一旦发现后端实现它,就**完全替换**本地 limiter(`scheduler.go:151`)。于是 5.4.3 倾向的"SQL 后端用数据库共享计数实现配额"和"继续走 `LimiterProvider`"**互不兼容**——SQL 后端若为了配额实现这个接口,就被迫连 Workers/RPS 一起接管:要么用数据库行锁做高频信号量(MySQL/PG 方案第 8 节已论证不划算),要么在实现里重造"这两个维度退回进程内"的行为。此外还有一层张力:能力接口模式的默认行为是**静默退化**,而配额恰恰是 5.4.3 论证的"退化不可接受"的能力。

**research 必须明确比较四条路,选一条再定接口**:

| 候选 | 做法 | 初步观察 |
|---|---|---|
| ① 独立 `QuotaProvider` 能力接口 | 配额单独一个可选接口,与 `LimiterProvider` 平行 | 改动最小;但"两个能力接口 + 各自的退化规则"要写清组合矩阵,且配额的退化必须是 fail-fast 不是静默 |
| ② 把 limiter 拆成 slot/rate/quota 三个 provider | 细粒度能力,后端按维度声明 | 结构最干净;动现有 `LimiterProvider` 的公开形态,redisbroker 要跟着改 |
| ③ 扩展 Broker 原子认领能力 | "检查+扣减+Dequeue"进同一个后端原子操作 | 即 5.4.2 的"精确认领配额"路线,破坏 Broker 接口的后端无关性,代价最大 |
| ④ SQL 后端实现完整 `LimiterProvider` | 连 slot/RPS 一起用数据库实现 | 高频信号量走行锁,MySQL/PG 方案第 8 节已论证不划算,基本可排除,列出只为完整 |

以下三小节是否决理由与必须满足的约束,四条路里选哪条都要过这三关。

#### 5.4.1 ⚠️ v1 接口为什么不成立(评审 #5,P1)

**v1 的 `CommitQuota()`(签名见附录 A.1)没有 `ctx`、没有返回值,根本无法表达"Redis 断连了"或"提交失败"。** 而 Dequeue 成功之后配额提交失败时,三条路都是死的:

- **忽略失败** → 少计 → 超发(而且是静默的)
- **同步无限重试** → 卡住一个已经认领的任务
- 此时 **handler 心跳还没起来**,卡久了租约会过期 → 任务被 reaper 回收 → 白白 `LeaseLost` 一次

**另外 v1 草案的 `INCR` + `EXPIRE` 分两条命令有崩溃窗口**:`INCR` 成功、`EXPIRE` 前进程崩 → 这个 key **永远没有 TTL** → 配额窗口再也不重置 → **该队列被永久锁死**。约束:redis 侧无论最终接口长什么样,必须用**单条 Lua 脚本原子完成计数和过期设置**(redisbroker 本来就有 `sem_acquire.lua` 的先例)。

#### 5.4.2 必须先定产品语义,再定接口(评审 #5)

初稿在 5.5 把"扣减时机"列成了 research 的头号问题,但**问题问错了层次**——那是实现问题,前面还有个产品问题没答:**这个配额到底是硬的还是软的?**

| 语义 | 承诺 | 实现路子 | 代价 |
|---|---|---|---|
| **硬配额** | 宁可少用,**绝不超发** | Dequeue **之前**用 Lua 原子扣减 | 接受空队列/进程崩溃造成的**保守损耗**(扣了没用上) |
| **尽力配额** | 允许超发,但**上界明确且可测** | 检查在前、扣减在后 | 要给出**形式化的超发上界**并写进合同——上界不是"进程数"(见下方修正) |
| **精确认领配额** | 不白扣也不超发 | 检查+扣减+Dequeue 必须在**同一个后端原子操作里** | **破坏 Broker 接口的后端无关性** |

**初稿的方案(检查在前、扣减在 Dequeue 后)其实是"尽力配额",但初稿把它当成硬配额在卖**(5.1 说的是"别超月底账单")。这两件事必须对齐:

> 当前接口不能同时满足"不白扣、不超发、故障可恢复、后端无关"。**四选三,必须先裁决。**

~~倾向:先做尽力配额,把超发上界写死进合同。~~ **裁决(2026-07-16)推翻了这个倾向:定为硬配额**——允许超发就不能宣传成"防止超支"的配额;保守损耗是接受的代价,不能接受就改名 `AdmissionBudget`。见第 13 节 #2 与《Quota 领域模型》。

**⚠️ "超发上界 ≈ 进程数"不是可靠合同(第三轮评审 #2,v4 修正)。** 真正的边界是"**同时处于检查与扣减之间的认领循环实例数**"——即 Gate 实例 × 该队列的 claim loop,不是部署进程数:一个进程可以创建多个 Gate;重连、超时会让"处于窗口中"的实例数瞬时膨胀;时钟偏差(5.3)还会在窗口边界叠加双放。**要把它写成公开保证,必须先给出并发模型和形式化上界**(哪些参与方、每个最多贡献几次超发、故障时的界),不能用"进程数"这种部署近似——这是尽力配额进 spec 前的硬前提,列入 research 必答。

#### 5.4.3 ⚠️ 多进程下"退回进程内配额"不可接受(评审 #6,P1)

初稿写"SQL 后端第一期不实现 `LimiterProvider`,自动退回进程内配额——和 RPS 的处理一致"。**这个类比是错的,而且现在错得更厉害。**

**(a) 现状比初稿写的更糟**:`pgbroker` / `mysqlbroker` **已经落地**,而且 `pgbroker/broker.go:6` 明确写着「LimiterProvider 不实现,scheduler 自动退回进程内限流」。这两个后端在 README 的定位是**多进程/多机**——不是 sqlite 那种同机多进程,是**跨机器、进程数无上界**。

所以"退回进程内配额"的实际效果是:10 个 pod 接同一个 Postgres,每个配 `QuotaLimit=5000` → **实际放行 50000 次**。

**(b) "RPS 也是这么退化的"不能作为理由——失效模式根本不同**:

| | RPS 退化 | 配额退化 |
|---|---|---|
| 后果 | 打网关的速率变成 N× | 总调用量变成 N× |
| 有没有可见信号 | **有**。打快了网关返 429 → `Throttled` 计数立刻涨,用户马上看得见、能手动调低各进程 RPS | **没有**。没有任何反馈信号,你只是默默超支 |
| 用户何时发现 | 立刻 | **月底收账单** |

(v4 修正:此表初版写过"RPS 退化会通过 429 自纠正/系统自己收敛"——**这个论据不成立,删除**。`ErrThrottled` 只是把**当前任务**退避重排,不会降低本地 limiter 的 RPS;只要队列里还有别的 pending 任务,各进程仍持续按各自配置的速率发请求。429 减轻的是单任务的重试频率,不构成总速率的收敛机制。两者的真实差别只在"有没有立刻可见的信号"。)

RPS 退化是"性能不精确但看得见",配额退化是"**钱花超了且无人告知**"。**这个功能存在的全部理由就是"别超账单",退化成 per-process 就等于这个功能不存在**——比不提供更坏,因为用户以为自己有保护。

**(c) 必须选一个并写进合同:**

1. **只有 redis 提供集群级周期配额**,其他后端明确是 per-process,**且功能名和 README 不得宣称是网关硬配额**
2. **SQL 后端用数据库共享计数实现**(一张 `quota` 表 + `INSERT ... ON DUPLICATE KEY UPDATE` / `ON CONFLICT DO UPDATE`)。初步判断行锁成本可接受(相对 RPS 信号量它每任务只走一次)——**但这只是初步判断,不是结论**:这条更新在每次认领的热路径上,且**所有消费者争同一行**(每窗口一行),争用行为必须压测后定论(v4 补:第三轮评审次要项)
3. ~~拿不到共享配额就拒绝该配置(`New()` fail-fast)~~ **v4 撤回:这条无法自动实现**——一个进程调 `New()` 时**不可能知道**有没有别的进程打开着同一个 sqlite 库文件,"仅多进程时报错"是判定不了的。现实选项只有三个:①sqlite 一律拒绝配额配置;②加显式的 deployment scope 配置让用户自己声明单进程/多进程(挡不住误配,只是把责任交出去);③**sqlite 也做库内共享计数**——配额表就建在同一个库文件里,跨进程天然共享,和 pg/mysql 走同一条路

**倾向 2 + ③**:sqlite/pg/mysql 全部用各自库里的共享计数表(同一套 SQL,`internal/sqlbroker` 与 sqlitebroker 各自落地),redis 用 Lua,memory 本来就只有单进程形态——**每个后端的配额都是其介质范围内可信的**。这样"配额"这个词在任何后端上都不说假话。

**✅ 裁决(2026-07-16)确认此方向**并补一条:不支持共享计数的后端(将来若有)**显式拒绝配置,绝不静默退回进程内**;各介质性能靠基准测试定论。见第 13 节 #3 与《Quota 领域模型》。

### 5.5 ⚠️ research 阶段的头号问题:扣减时机

`claimLoop` 现在的顺序是**先占槽 → 再等令牌 → 最后 Dequeue**。配额插在哪里,以及**何时扣减**,是这一项唯一的设计难点,必须在 research 阶段解决,不能想当然。

问题的核心:**检查配额和真正认领到任务之间有窗口**。

- 如果在 Dequeue **之前**扣:队列空了没认领到任务,配额白扣了。要退还就是补偿事务,多进程下容易漏(进程崩在中间 → 配额永久泄漏)
- 如果在 Dequeue **之后**扣:任务已经拿出来了,此时发现超配额只能 Requeue,污染状态、且可能反复空转
- 现有的 `WaitToken` 也有同类的白烧问题,且注释里如实承认了(v4 按第三轮评审 #6 修正引用):实际顺序是 **AcquireSlot → WaitToken → Dequeue**(`scheduler.go:248`「顺序写死:先占并发槽,再等 RPS 令牌,最后才 Dequeue」),白烧发生在**拿到令牌之后 Dequeue 扑空或出错**时(`scheduler.go:262`「队列空闲期会预烧至多 1 个令牌等在 Dequeue 上,Dequeue 出错也白烧 1 个」);"先拿令牌再等槽会白烧"是注释里解释**为什么不采用**那种顺序的反例,不是现状

**但配额和令牌的容错度不同**:令牌白烧只是让实际 RPS 略低于配置(保守方向,无害);**配额白扣会让用户少调 N 次网关**,如果周期是一天,白扣就是真金白银的损失,而且用户看不出来为什么。

~~初步倾向:"只检查不扣"放占槽前、"真扣"放 Dequeue 后(尽力配额时序)。~~ **裁决(2026-07-16)选了硬配额,此时序作废**:扣减/预留必须在**认领之前**由后端**原子**完成("检查窗口 + 扣减"一步),接受"预留了没用上"的保守损耗;认领扑空尽力退还,退不掉当泄漏(保守方向)。完整的 `QuotaReservation` 生命周期见《Quota 领域模型》第 2 节;空队列轮询下的白烧节奏是该文档问题清单 #2。

### 5.6 验收(评审 #12 修订:初稿放错了层)

## 6. 改进项 2(⛔ 退回 research,性质是补漏):终态任务的 ID 永久占死

> **本节现行立场(v3 收口)**:**需求成立**(6.1 的 cron 配方死锁 + "清理交给使用方但没给工具"),**设计未定**——v1 的 `Delete` API(签名、约束、6.4.1 拦截判据)已被"ID 复用静默篡改历史依赖指向"这一发现整体否决,存档见附录 A.2。往下推进要先过第 13 节的裁决 #1:任务身份语义,五个候选见 6.4。

### 6.1 现象

README 给的 cron 配方(方案 v5 第 10 节,非目标"cron 周期调度"的替代路子):

> 应用侧 robfig/cron 触发 Submit + 确定性任务 ID(如 `daily-report:2026-07-14`)靠 `ErrTaskExists` 幂等去重,不需要选主和分布式锁。

**这个配方在任务失败后会死锁。** 已核实:

1. **同 ID 拒绝是无条件的,不看现有任务什么状态**:
   - `memorybroker/broker.go:168` —— `if _, exists := b.recs[id]; exists { return ErrTaskExists }`
   - `sqlitebroker/enqueue.go:30`
   - `redisbroker/broker.go:240`
2. **全库没有任何删除 API**:`broker.go` / `client.go` 里 grep 不到 `Delete` / `Purge` / `Remove` / `Cleanup`
3. `brokertest` 契约 2(`IdempotentID`,`cases_basic.go:97`)把这个行为**焊死了**:"同 ID 二次入队 → ErrTaskExists,且原任务原封不动"

后果:今天的 `daily-report:2026-07-16` 如果重试耗尽进了 `failed`,这个 ID **永久占死**——不能重提、不能删除。明天的 cron 会触发新 ID(`:2026-07-17`)所以不受影响,但**今天这份报告再也跑不出来了**,除非人工去 SQL / redis-cli 动数据。

而非目标清单里写着"结果长期存储 / TTL 管理 ❌ 清理交给使用方"——**但使用方手上没有清理的工具**。这两条凑在一起就是个洞。

### 6.2 参照:funboost 踩过这个坑

funboost 的任务去重(`consumers/redis_filter.py` + `do_task_filtering`)是按函数入参 md5 生成指纹去重的。它的关键设计:**成功后才写去重指纹**(`base_consumer.py:919-920`,在 `_run` 的 finish 分支),**失败的任务不会被后续误过滤**。

它还有一个配套细节:requeue 模式的重试消息会被 `_should_filter_task` **特意放行**(`base_consumer.py:868`),否则重试消息会被去重过滤器误杀。这是踩坑后的补丁。

**注意:不建议照抄它的"按入参自动 md5 去重"** —— taskgate 的显式 `WithID` 更清楚,不用猜指纹怎么算、不用处理"URL 里的随机参数干扰"这类问题(funboost 为此又加了 `filter_str` 让用户手动指定指纹字段)。这里要借鉴的**只是"终态是否应该释放标识"这个问题意识**。

### 6.3 v1 的三个解法选项(记录备查;现行候选清单在 6.4)

| 选项 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| **A. 终态 ID 可复用** | Enqueue 时若同 ID 任务已是终态,允许覆盖 | 配方无需改动,用户无感 | Enqueue 语义变复杂(同 ID 有时拒绝有时覆盖);**旧任务记录被覆盖 = 丢历史**;动 brokertest 契约 2,四份实现全改(memory/sqlite/redis/sqlbroker) |
| **B. 加 `Delete` API** | `Delete(ctx, id) error`,只允许删终态 | 不动契约 2,回归面小;**顺带解决"清理交给使用方但没工具"**这个更大的问题;语义单一 | 配方要多一步(Submit → ErrTaskExists → Get 看状态 → failed 就 Delete 再 Submit),稍绕 |
| **C. `WithID` + `ReplaceIfFinal()` 选项** | 显式声明"终态就覆盖" | 语义显式,不影响默认行为 | 仍然丢历史;仍要动四份实现的 Enqueue;新增一个 SubmitOption 只服务一个场景 |

**v1 曾建议 B 并给出了具体 API 与拦截判据——该设计已被 6.4 的身份语义问题整体否决,草案存档附录 A.2。** 当时选 B 的四条理由记录如下备查(其中 1、2、4 在裁决时仍有参考价值):
1. **不动既有契约**。契约 2 的"同 ID 二次入队 → ErrTaskExists,且原任务原封不动"是个好合同,A 和 C 都要削弱它
2. **解决的问题更大**。非目标清单说"清理交给使用方",但没给工具。`Delete` 一并把这个补上——用户能清理旧任务、能释放 ID,是同一个 API
3. **语义单一**:删除一个终态任务。没有"有时拒绝有时覆盖"的分支
4. **不会因 Submit 的隐式覆盖而丢历史**——删除本身仍然是显式、不可逆的历史清理操作(评审 #13 修正:初稿写的是"不丢历史",那是**美化**。`Delete` 只是把"何时丢历史"的决定权交给用户,不是不丢。这个措辞会掩盖审计、依赖溯源和 ID 复用问题——正是 6.4 那个洞被漏掉的原因之一)

### 6.4 任务身份语义(待裁决,第 6 节的阻断项)

> v1 在这里给出过 `Delete` 的完整 API 签名、约束清单和"只拦未终态子任务"的拦截判据(借 dask `local.py:290-317` 的引用计数推出),**已整体否决,全文存档附录 A.2**。否决原因和错误根源保留在下面——它们框定了任何后续设计。

**v1 判据的错误根源**:把 dask 的瞬时图节点直接类比成了 taskgate 的持久化任务历史——这两个东西的生命周期语义根本不同:

- dask 的 `waiting_data` 管的是**一次 `compute()` 之内**的中间数据。计算结束,整个 `state` 连同 `cache` 一起丢掉,没有"历史"这个概念,更没有"key 被复用"的问题
- taskgate 的任务是**持久化的历史记录**,而且删除之后**同一个 ID 可以被重新提交**——这正是第 6 节要解决的 ID 占死问题的另一面

于是 v1 判据会制造出这个场景:

1. `P(v1)` 完成
2. `C` 完成,且 `C.DependsOn = ["P"]`
3. `C` 已终态 → 按 v1 判据,**允许删 `P(v1)`**
4. 用同一个 ID 提交 `P(v2)`
5. 此后任何人按 `C.DependsOn[0]` 去 `Get`,**拿到的是 `P(v2)`,不是 `C` 当年真正依赖的那个父任务**

即**删除 + ID 复用会静默地篡改历史依赖的指向**。对一个"任务状态可查、可追溯"为卖点的库,这是不可接受的。

**必须在进 spec 前,先裁决任务身份语义。六个候选(v3 补入 e,v4 修正 e 并补入 f):**

| 选项 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| **a. 有任何 dependent 记录就禁止释放 ID** | 退回最粗的判据 | 简单、安全 | `Delete` 对有子任务的父任务基本失效,cron 配方那个洞只在"无子任务"时被堵上 |
| **b. 删除时级联删掉完整依赖历史** | 删 `P` 连带删 `C` | 无悬空引用 | 删一个父任务把整条流水线历史删了,破坏性大,且要防"删到一半崩" |
| **c. 引入不可复用的内部 generation** | 依赖引用 `(id, generation)` | 历史精确、ID 可复用 | 动 `Task` 结构、`task_deps` 结构、`DependsOn` 语义,五后端全改 |
| **d. 拆成两个概念/API** | "清理任务数据" 与 "释放业务幂等键" 分开 | 各管一件事,语义清楚 | 要重新想清楚 cron 配方到底需要哪个 |
| **e. 原地重跑(`Retry(ctx, id)`)** | **保留同一任务身份**,把终态任务重新置回 `pending`;不删数据、不释放 ID、不发生 ID 复用 | 直接对准"这份报告再跑一次";无悬空引用;不动 Enqueue 语义和契约 2;改动面小 | **避免了重绑定,但没保住历史(v4 修正)**:`Task` 只有一份可变的 `Result`/`LastError`/计数/时间字段(`taskgate.go:67`),重跑后旧子任务按父 ID `Get` 到的是**新结果**,不是它当年消费的那份——e 解决的是"重新执行同一个作业",**不是**"保存执行历史"。若按 Retry 收窄(只允许 `failed` 任务、延续同一次 execution、延续计数),这个缺陷可以接受(失败任务没有被消费过的旧结果);若允许重跑 `completed` 任务,历史篡改问题换了个形式回来 |
| **f. 稳定业务键 + 不可复用 execution ID + `replay_of`(v4 补入,第三轮评审指出的最常见模型)** | 把"业务幂等键"(用户的 `daily-report:2026-07-16`)和"任务身份"(一次 execution 的不可变 ID)**拆成两个概念**:Replay 创建**新 execution**、记 `replay_of` 指回旧的;依赖永远引用不可复用的 execution ID;业务键上的幂等约束单独建模("同一业务键最多一个非终态 execution") | **历史完全不可变**——旧 execution 的 Result/计数/依赖关系永不被改写,审计与溯源天然成立;ID 复用问题从概念上消失;Retry(e 的收窄版)可以作为"延续当前 execution"的轻量补充共存;**是任务系统里最常见、最经得住时间的模型** | 是 d 和 c 的合体落地:动 `Task` 结构(加业务键字段或 `replay_of`)、Submit 幂等判定逻辑、五后端存储与索引,改动面最大;`DependsOn` 引用的是业务键还是 execution ID 要想清楚(流水线配方受影响) |

**✅ 裁决(2026-07-16):采纳 f,e 暂缓。** f 是目标模型;e 只有在明确限定为 failed-only、且不覆盖历史的情况下才考虑,**否则不单独落地**——原因是 e 会把"同一任务身份对应多次结果"的临时语义写进公共 API,之后迁移到 f 还要背兼容。短期 cron 需求由业务层生成新 execution ID 并另存业务键;若急需重跑 API,做**明确命名的 Replay,不叫 Retry**(Retry 保留给同一 execution 内的失败重试)。四概念的完整定义、Submit/Replay 规则与影响面见《[2026-07-16-Identity领域模型.md](./2026-07-16-Identity领域模型.md)》。

**在此裁决前,第 6 节整体不进 spec。**

### 6.5 影响面(评审 #4 修订:初稿严重低估)

> 本节按"若 6.4 最终仍走通用 `Delete`"评估——它证明的是这条路**远比 v1 想的贵**,这本身是裁决 #1 的重要输入。若裁决走"原地重跑"(选项 e),下面 (a)(b) 的悬空引用问题整个不存在(ID 不释放),影响面另行评估。

**初稿把 SQL 侧的改动描述成"很小:多实现一个 `DELETE WHERE id=? AND status IN (终态)`"。这是错的。** 核实后:

**(a) 悬空引用有两处,不是一处**

```sql
-- sqlitebroker/schema.sql
CREATE TABLE task_deps (
    child_id  TEXT NOT NULL,
    parent_id TEXT NOT NULL,
    done      INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (child_id, parent_id)
);   -- ← 没有 FOREIGN KEY,没有 ON DELETE CASCADE
```

只删 `tasks` 一行会留下:
1. `task_deps` 里 `parent_id = 被删ID` 的边
2. `task_deps` 里 `child_id = 被删ID` 的边
3. **`tasks.depends_on` 这一列**——`DependsOn` 还以 JSON 数组文本**在别的任务行里又存了一份**(`sqlitebroker/enqueue.go:89`:「DependsOn 原样存 JSON 数组文本(往返一致),去重只作用于 `task_deps`」)。评审只点到了 `task_deps`,**实际还有这第三处**

ID 复用后,残留的旧边会**参与新任务的依赖传播**(`propagate.go:27` 的 `UPDATE task_deps SET done=1 WHERE parent_id=?` 会扫到它们)。

**(b) `Delete` 必须原子完成的动作清单**

- 校验任务存在且为终态
- 校验 dependents 是否允许删(判据待 6.4 裁决)
- 删/改 `task_deps` 里 `child_id = id` 的边
- 删/改 `task_deps` 里 `parent_id = id` 的边
- 处理其他任务 `depends_on` JSON 列里对该 ID 的引用
- 清 memory 的 `children` 切片、redis 的反向索引
- 同步扣 `Counts`
- **与并发 `Enqueue(DependsOn(id))` 正确串行化**

**(c) 并发竞态是真的**

MySQL/PG 下:事务 A 正在 `Enqueue` 一个 `DependsOn(P)` 的子任务(已校验 P 存在),事务 B 同时 `Delete(P)`。两个事务都提交 → **子任务引用了一个不存在的父任务**,而且它会永远 `blocked`(没人会来唤醒它)。这需要显式的锁序或约束,不是"加个 DELETE 语句"能解决的。

**(d) 后端数量:三 → 五**

**本方案初稿写的"三后端"已经过时。** `pgbroker` / `mysqlbroker` / `internal/sqlbroker` 已落地(工作树里是 untracked 新目录,README 已改成"五种后端一套契约")。所以:

- `Broker` 接口 15 → 16,要改的是 **memory / sqlite / redis / sqlbroker(pg+mysql 共享核心)四份实现**
- **`docs/plans/2026-07-15-MySQL-PG后端适配方案.md` 的"结论先行"第 1 条「Broker 接口一行不用改」已经被本方案打破,而且那个方案已经落地了**——这不再是"若已开工要协调"的假设,是**既成事实,必须回头修订该文档**
- `brokertest` 新增的契约要五后端全过,其中 pg/mysql 走环境变量门控 + CI service container

**(e) 验收要补**(评审给的清单,照收)

- 删除后复用 ID 不受旧依赖边污染
- 删除父任务与并发提交子任务不产生悬空引用
- 删除作为 child 的任务后,旧父任务终态传播不影响复用后的新任务
- `Counts` / `List` / `Get` / 依赖传播四处口径一致

- 新增导出符号:`Delete`、`ErrNotFinal`、`ErrHasDependents`,godoc 要齐

### 6.6 为什么建议单独提

这一项**性质和"加新功能"不同,是补既有设计的漏**。方案 v5 定 cron 配方时没想到"失败后 ID 释放"这一层,是设计遗漏。

建议单独开 spec、单独审查,不要和周期配额混在一个里程碑——两者的风险等级不一样(配额是纯增量,`Delete` 动 Broker 接口)。

### 7.2 事实:taskgate 是 at-least-once,而且是设计使然

README 自己在别处写得很清楚,handler **会被重复执行**:

- `README.md:20`(特性):「worker 崩溃后 reaper 把任务**捞回重跑**」
- `README.md:131`(长任务一节):「卡死就停止续租,租约到期被 reaper **回收重跑**」
- M3 完成记录:Shutdown 超时 → `Requeue` 归还回 pending → 后续**重跑**(且三计数一个不动)

所以 handler 执行是 **at-least-once**,不是 exactly-once。

**"恰好执行一次"想表达的应该是"同一任务不会被两个 worker 同时认领"**(Lua 原子流转保证互斥),这个说法本身是对的、也是 M2 的真实成果。但 **exactly-once / 恰好执行一次是个有确切含义的术语**,指副作用只发生一次——而租约回收明确违反它。用这个词描述互斥性是误用。

### 7.3 为什么这条对 taskgate 特别严重

taskgate 的目标场景写在 README 第一段:「调用 LLM / OCR 这类**有配额的外部网关**」。这类 handler:

- **花钱**:每次调用产生账单
- **有副作用**:往库里写结果、发通知、扣额度

用户读到"恰好执行一次"→ 合理地不做幂等防护 → worker 崩在 LLM 调用中途 → 租约回收 → 任务重跑 → **账单翻倍、数据重复写、通知重复发**。

而且**全库没有一处告诉 handler 作者"你必须幂等"**。README 从使用者视角讲了租约会重跑(那是特性描述,读起来像好事:"崩了也不丢任务"),但从没讲**这对写 handler 的人意味着什么义务**。

### 7.4 dask 的标准修法:"Task Expectations" 契约节

`ref/dask/docs/source/graphs.rst:111-143` 有一节 **Task Expectations**,做法是三件套:

1. **陈述契约**:Don't Modify Data In-Place
2. **说清为什么,且直接和重跑挂钩**(原文):

> *"...if other tasks need to use `x`, or **if Dask needs to rerun this computation multiple times because of worker failure**."*

3. **给具体反例**:`arr[arr > 5] = 0` 会污染其他 task 看到的 `x`

即:**把"为什么必须无副作用"和"我们会重跑"直接绑在一起写进契约文档,而不是藏在 FAQ 里**。有了 retry / 崩溃回收,就是 at-least-once,就必须把这个约束提升成对 handler 作者的显式契约。

### 7.5 建议改动(纯文档,不动代码)

**(a) 改掉三处"恰好执行一次"措辞**(中英各三处,见 7.1 表)

改成准确的说法。要表达的是互斥性,不是 exactly-once:
- 后端表:「多进程/多机、**同一任务不会被两个 worker 同时认领**」
- Redis 后端一节:说清 Lua 原子流转保证的是**认领互斥**,以及崩溃后会重跑 → **at-least-once**
- 里程碑:同步改

**(b) 新增一节「handler 的执行契约」**,建议放在现有「handler 的错误语义」(`README.md:112`)之前,两节都是面向 handler 作者的契约,应当相邻。内容:

- **taskgate 是 at-least-once,handler 必须幂等**——一句话说死,放在最前面
- **列出会导致重跑的三条具体路径**(这是用户最需要知道的,因为每条都不是"出错了"而是正常运转):
  1. worker 进程崩溃 → 租约到期 → reaper 回收重跑(`LeaseLost` 计数)
  2. handler 卡死 + `ManualHeartbeat` 档停止续租 → 租约到期回收重跑
  3. `Shutdown` 超时打断 → `Requeue` 归还 → 重跑(且三计数一个不动,任务看起来"没跑过")
- **给 LLM 场景的具体反例**(对标 dask 给 `arr[arr > 5] = 0`):handler 里"调 LLM → 扣用户额度 → 写库",崩在写库前 → 重跑 → 额度扣两次
- **给防护配方,但必须把"本地副作用"和"外部调用"分开讲**(见 7.5.1、7.5.2 两条修正)
- **说清 `WithID` 幂等的是"提交",不是"执行"**——这两个是不同的东西,现在 README 只讲了前者(`README.md:124`、`:263`、`:296` 都只说 `WithID` 幂等去重),用户容易以为"我用了 WithID 就幂等了"

#### 7.5.1 ⚠️ 撤回:不得建议用 `OnStateChange` 承载关键副作用(评审 #1,P0)

初稿的防护配方里有一条:"把扣额度这类不可重放的操作挪到 `OnStateChange` 回调里按终态做一次"。**这条是错的,已从 README 删除,方案同步撤回。**

核实结果——`OnStateChange` 根本承载不了关键副作用:

- **合同就是异步通知**:`broker.go:14` 写死「`Notify func(Task)` 状态流转回调,**可 nil**;**必须在锁/事务外异步调**」
- **panic 被静默吞掉**:`memorybroker/broker.go:109`、`sqlitebroker/broker.go:160`、`redisbroker/broker.go:204` 三处都是 `defer func() { _ = recover() }()`——回调里出错,**没有任何人会知道**
- **没有送达保证、没有顺序保证、没有重试、没有"全局只消费一次"**:多进程下每个进程都注册了自己的回调;进程崩在回调里,这次通知就永远丢了

所以它既可能**漏执行**(回调 panic 被吞、进程崩在回调前 → 通知丢了 → 额度没扣),也可能**重复执行**(任务重跑,每次流转都再触发一遍 → 额度扣多次)。(v4 措辞修正:初版写"多进程各跑一次"不准确——回调只在**执行该状态流转的那个进程**本地触发,不是广播;真正的问题是通知可能丢失、可能随重跑重复,且与状态落库不在一个事务里,无法保证送达。)**用它扣费比不做防护还危险**——因为它看起来像是做了防护。

**正确的定位写死**:`OnStateChange` 只用于埋点,以及允许丢失/重复的观测逻辑。关键副作用必须在 handler 内、用 `Task.ID` 配合数据库唯一约束 / 幂等表 / 事务性 outbox 原子完成。

#### 7.5.2 ⚠️ 修正:本地幂等消不掉"重复调用外部网关"(评审 #2,P1)

初稿的"正确示例"是:`callLLM` → 用 `Task.ID` 原子保护扣费和落库,并宣称这样就不会"LLM 调两次、额度扣两次"。**前半句是假的。**

进程若崩在 `callLLM` **成功之后**、`SaveWithIdempotencyKey` **之前**,重跑必然再调一次网关——**钱照花**。本地幂等键保护的只是**本地账务和落库不重复**,消不掉外部调用的重复。

初稿的示例在讲一个 at-least-once 的诚实契约,自己却在这里悄悄宣称做到了 exactly-once 的效果,**这正是第 7 节要纠正的那种毛病的复发**。

**README 已改成分三类讲**(v4 同步:初版此表把"发通知"归进了本地事务可防的一列,恰好复发了本节要纠正的毛病):

| | 本地数据库副作用(扣额度、落库) | 外部网关调用(调 LLM/OCR) | 异步消息/通知(邮件、Webhook、短信) |
|---|---|---|---|
| 能不能防住 | **能**,`Task.ID` + 唯一约束/幂等表 | **取决于网关**,taskgate 无能为力 | **只能防到"意图不丢不重"**,投递仍是 at-least-once |
| 手段 | 扣费与落库必须在**同一事务、同一唯一键**内完成(分两步做仍有窗口) | 网关**明确支持** idempotency key 时把 `Task.ID` 传过去,这是**唯一**能真正消除重复计费的办法(支付类网关普遍支持;具体 LLM 网关/端点是否支持以官方文档为准,本方案不替它们承诺);不支持就只能接受成本或在业务协议层补偿 | 事务性 outbox 把"要发这条通知"随业务事务落库;**投递端/接收方仍要按幂等键去重**,outbox ≠ 不会重复通知 |

并明确写出:**"外部调用已成功、但本地尚未落账"这个不确定窗口,任何任务队列都消不掉**,taskgate 不例外。

**(c) 顺带明确"我们不做 exactly-once"**

dask 的 `faq.rst:255-262` 用"我们不是什么"来定义架构(*"We do not use Spark-style shuffle systems. We do not use MPI-style collectives."*),信息密度极高。taskgate 的非目标清单可以加一条:**不做 exactly-once 执行**——这是分布式系统的老大难,库层面给不了,只能靠业务侧幂等键。写明比不写好。

### 7.6 为什么这条排在最高优先级

- **不动代码,不动契约,零回归风险**——就是改文档
- **但现在这个措辞在主动误导用户往花钱的坑里走**,而且踩了不会立刻发现(崩溃重跑是低频事件,账单对不上时早就过去了)
- 其他三项都是"缺个能力",这项是"**说了假话**"

## 8. 分析后不建议做的项

这些初看有价值,深入分析后收益不成立。记录分析过程,避免将来重新想一遍。

### 8.1 `Wait()` 改事件驱动(funboost)—— 收益最大的场景恰恰是最贵的实现

**现状**:`Wait()` 是 50ms 轮询 `Get`(`client.go:253`,`waitPollInterval`)。

**funboost 的做法**:consumer 把结果 `LPUSH` 到以 task_id 为键的 Redis list 并设 TTL,发布端用 `BLPOP(task_id, timeout)` 阻塞等待(`core/msg_result_getter.py:79-87`)。拿到后再 `LPUSH` 回去 + `EXPIRE`,让结果可重复读。用 list+blpop 而不是 pub/sub,天然解决"结果先到、等待者后来"的竞态。零轮询。

**初看的收益**:redis 后端下每个等待者每秒 20 次 HGETALL,等的人一多就是可观的无谓流量。

**深入分析后收益不成立**:

把 `Wait` 拆成两种场景:
- **本进程等本进程在跑的任务**:可以挂 `OnStateChange`(`BrokerOptions.Notify` 已经透传),改动很小
- **纯生产者进程等别的进程在跑的任务**:必须跨进程通知,要给 Broker 加新能力接口(`ResultWaiter` 之类)+ redis 侧实现 blpop 通道 + 结果写入通道的时机/幂等/TTL 一整套

问题在于:**最需要优化的恰恰是第二种**。"提交后等结果"这个动作,天然发生在生产者进程,而生产者进程往往不跑 worker。第一种场景(本进程等自己在跑的任务)反而少见——都在一个进程里了,直接调函数就好,何必过队列。

所以:便宜的那一半没什么用,有用的那一半要动 Broker 接口(和 4.7 一样的地基问题),成本和周期配额相当,但解决的只是"轮询有点浪费"。

**建议不做**,除非出现实测的轮询压力(比如"上千个 `Wait` 并发把 Redis 打满")。真出现了再评估,那时也有实测数据支撑设计。

### 8.2 `Task.Metadata`(funboost)—— 为还不存在的需求开口子

**问题陈述**:taskgate 明确没有 metrics / trace 集成(非目标)。一般来说"用户拿 `OnStateChange` 自己接"就够了。但 OTel 有个特殊之处:

funboost 的 OTel 集成(`contrib/override_publisher_consumer_cls/funboost_otel_mixin.py`)关键在**把 trace context 注入消息体 `msg['extra']['otel_context']`,消费端取出来 extract 成父 span**。没有这一步,消费者生成的 span 就是孤立 trace,失去追踪意义。

taskgate 的 `Payload` 是用户的 `json.RawMessage`(库不该往里写东西),`Task` 结构上也没有 `Metadata map[string]string` 这类字段。所以**跨任务的链路追踪是用户自己补不了的**——而 taskgate 恰好有依赖流水线(检索→生成→打分),这正是最想看全链路的场景。

**这是本次调研里唯一一个"用户自己补不了"的可观测性缺口,所以值得记录。**

**但建议先不做**,理由:
1. **没有诉求方**。和优先级调度、webhook 一样,是"想得到"而不是"有人要"
2. **开口子的代价是滑坡**。funboost 的 `extra` 字段就是前车之鉴:一开始只是放点元信息,最后塞满了 `eta` / `countdown` / `_advanced_retry_count` / `otel_context` / `filter_str` / `can_not_json_serializable_keys`,变成第二个 payload,且每个字段都要在 `_get_priority_conf` 里做优先级回退。一旦 `Metadata` 存在,后续每个"要不要加个字段"的讨论都会指向它
3. **有替代路子**。用户可以在自己的 payload 里包一层 `{trace: ..., data: ...}`,虽然丑,但能用

**如果将来要做,该怎么做**(免得重新想):
- 字段:`Metadata map[string]string`,库**只读不写**,原样透传并落库
- 边界写死在 godoc:这是给跨切面数据(trace context 之类)用的,**不是给业务数据用的**,业务数据进 `Payload`
- 严禁库自己往里塞字段——一旦库开始用,就是 `extra` 的老路
- 五后端要落这个字段,`brokertest` 加一条"Metadata 原样透传"

**dask 的 `annotate` 是同一问题的另一个答案,但不适用**:它把 retries/priority/resources 挂成 annotation(`base.py:89-204`),定位是 *"metadata or **soft constraints** ... schedulers **may choose to respect**: They signal intent **without enforcing hard constraints**"*。本地调度器完全忽略它们,只有 distributed 才理会。这个"可以被无视"的定位让它不会滑坡成第二个 payload。**但 taskgate 要放的 trace context 不是建议性的,是必须原样保真的数据**——两回事,上面的分析不变。

### 8.3 LIFO ready 队列(dask)—— 目标函数相反,且会饿死老任务

dask 最出名的调度策略,`local.py:13-25` 文档化了 WHY:

> *"We preferentially choose tasks that were just made available in a **last-in-first-out** fashion... This results in more **depth-first rather than breadth first** behavior which encourages us to process batches of data to completion before starting in on new data."*

理由是 FIFO 会让大量 chain 同时开工、每条的中间结果都占着内存不能释放;LIFO 做完一条链再开下一条,中间结果尽早释放。

**不适用 taskgate,两条理由:**

1. **目标函数相反**。dask 的目标是最小化**内存**驻留(`scheduling-policy.rst:14-17` 明确写死这是单一目标函数)。taskgate 的结果存在 broker 的磁盘/Redis 里,而且**压根没有自动回收**(结果一直留着等用户 `Delete`),"尽早释放"这个收益根本不存在。
2. **公平性**。dask 的所有任务属于同一个用户的同一次 `compute()`,没有多租户、没有延迟 SLA,饿死谁都无所谓。taskgate 是任务队列,**LIFO 会让老任务永远排不上**。

#### 8.3.1 ⚠️ 撤回:初稿的"共享队列会先跑完所有检索"是事实错误(评审 #8)

初稿在这里写过一个"窄的真实场景":`Routes` 允许多 Type 共享队列时,FIFO 会导致"所有检索跑完才开始任何生成",首个结果延迟极差,并建议把这个后果写进 README。

**这个结论是错的,已核实,整条撤回。** 错因是**把 `List` 的查询排序当成了执行顺序**:

- `List` 的排序确实是 `(CreatedAt, ID)` 升序,但那只是**管理查询 API 的返回顺序**
- **Dequeue 根本不按 CreatedAt 选任务**。`memorybroker/broker.go:232` 的注释写死了:「取 `RunAt` 最早的(同刻取 ID 最小)」;sqlite 侧的索引也是 `idx_claim ON tasks(queue, status, run_at)`。**认领序是 `(RunAt, ID)`**

而子任务的 `RunAt` 是**入队时**就定下的(`memorybroker/broker.go:202-203`:零值时取 `now`),不是解除 blocked 时才定的。所以:

> 提交 doc1 的检索(RunAt=t1)、doc1 的生成(blocked,RunAt=t1),再提交 doc2 的检索(RunAt=t2>t1)。doc1 的检索跑完 → doc1 的生成转 pending,带着 **RunAt=t1 < t2** → **它会排在 doc2 的检索前面被认领**。

即**在这个具体提交模式下**(逐份文档提交父子任务、都不显式设 `RunAt`),doc1 的下游会排在 doc2 的检索之前——初稿担心的"所有检索先跑完"在该场景不成立。**但这只是该提交模式的结果,不是全局调度性质**:若先批量提交所有根任务、之后再补提交子任务(子任务的 `RunAt` 取的是它自己的提交时刻),或用户显式设置了 `RunAt`,认领顺序就完全不同。不要把它上升为"taskgate 天然深度优先"。

**教训记档**:`List` 的排序合同和 Dequeue 的认领序是两回事,不能互推。本方案初稿犯了这个错,后续讨论调度行为时必须引 `broker.go` 的认领实现,不能引 README 的 `List` 分页一节。

**因此**:8.3 的结论(LIFO 不适合 taskgate)**保留**,理由是目标函数相反 + 会饿死老任务这两条;但初稿附带的 README 改动建议**取消**。若将来真要讨论共享队列对首个结果延迟的影响,**必须先有确定性的调度测试或基准**,用实测数据支撑,不能靠推演。

### 8.4 `order.py` 静态优先级(dask)—— 已明确拒绝优先级,但记下将来若做的正确路子

dask 用 839 行(`order.py`)只为决定"多个任务同时就绪时先跑哪个",算出一个 `{key: int}` 全序当 tie-breaker。

**taskgate 已明确拒绝优先级调度**(方案 v5 第 11 节标"(可选)",M3 完成记录复核后维持不做,理由:唯一诉求方用多队列隔离已够,加优先级要动五后端的出队序合同)。**dask 有这个不构成重新讨论的理由**——它需要是因为一次 `compute()` 提交的是整张图、必须自己决定顺序;taskgate 的任务是逐个提交的,天然有 `CreatedAt` 序。

**但如果将来真要做,dask 有两条硬约束值得先记下:**

1. **静态预计算 + 运行时只做整数比较**(`order.py:26-39`):提交前算好优先级整数存进后端,dequeue 时只按整数排序。**绝不在热路径遍历依赖图**。五种后端都能高效支持"按整数取最小"(Redis ZSET / SQL 索引 / 内存堆),但没一个能高效支持运行时图遍历——这条直接决定 Broker 接口能不能保持窄。
2. **先写下目标函数和决策时间预算**(`scheduling-policy.rst:14-17`):dask 明确写"scheduling budget of **one millisecond per task, irrespective of the number of tasks**"。这一句话就砍掉了所有"扫一遍全队列找最优"的设计。taskgate 若做优先级,应当先写死"dequeue 决策 O(1),不随队列长度增长"。

### 8.5 5 个回调钩子(dask)—— taskgate 的结构本来就是对的

dask 的回调有 5 个钩子(`callbacks.py:15-23`):`start` / `start_state` / `pretask` / `posttask` / `finish`。而且它的 `Cache` 功能**完全构建在这 5 个公开钩子之上**(`dask/cache.py:12` —— `class Cache(Callback)`),没开后门。

这给了一个好的检验标准:**hook 系统够不够强,看能不能用它实现一个真功能**。

**但 taskgate 不需要加钩子**:
- `OnStateChange` 的定位就是**埋点**(方案 v5 第 10 节),它对这个定位是够的
- **限流没有塞进回调,而是走 `LimiterProvider` 这个独立的可选能力接口**(`broker.go:62-65`)——这才是对的结构。dask 把 Cache 塞进 Callback 是因为它没有别的扩展点;taskgate 有专门的能力接口,不该退回去用通用钩子
- 本方案里新增的周期配额、以及将来的熔断,都应当走**能力接口**这一类结构,不是加钩子(注意:不等于直接复用现有 `LimiterProvider`——它是整包替换的,配额复用它有粒度问题,见 5.4 的四路比较)

## 9. 挂起项(记录理由备查)

以下三项能说清价值,但都没有真实诉求方。按项目一贯的 YAGNI 纪律挂起,理由记在这里,避免下次调研重新讨论。

### 9.1 熔断(Circuit Breaker)

funboost 的 `contrib/override_publisher_consumer_cls/circuit_breaker_mixin.py` 是完整的三态机:
- CLOSED → OPEN(触发)→ HALF_OPEN(试探)→ CLOSED / OPEN
- 两种触发策略:`consecutive`(连续失败计数)/ `rate`(滑动窗口错误率,含 `min_calls` 最小样本数防止小样本误判)
- 两种计数后端:`local`(threading.Lock)/ `redis`(跨进程共享)
- 两种熔断行为:阻塞模式(消息留在 broker)/ fallback 降级模式

**taskgate 缺的是"队列级的集体刹车"**:网关整体挂了,现在的行为是每个任务各自 `ErrThrottled` 各自退避、`Throttled` 各自涨到 100 才进 `failed`。没有"整个队列停一会儿"的机制。

**但挂起**,理由:
- 现有 backoff(`min(2^n × 1s, 10min) ± 20%`)已经能把重试压到 10 分钟一次,对下游的压力其实有限
- 没有实际案例。等真出现"网关挂了半小时,taskgate 把它打得更惨"再说
- 如果要做,应当沿 `LimiterProvider` 的路子做成可选能力接口,不要往 `QueueConfig` 堆字段

### 9.2 队列暂停 / 恢复

funboost 的做法很朴素:Redis hash 存标志位(`funboost_pause_flag`,field=queue_name),消费循环每 10s 从 Redis 拉 flag,pause 时每 5s 轮询、**消息留在 broker 不被取出**(`base_consumer.py:619-627`、`:1228-1234`)。

**价值**:网关维护窗口时想让队列停一会儿,又不想停整个进程。

**但挂起**,理由:
- 和 taskgate 现在"不读 env、不读配置文件、只吃传进来的 `Config`"的干净模型**有张力**——它引入了"运行期改状态"这个新概念,而且状态存在库外(Redis),纯内存后端还没地方放
- 没有诉求方
- 部分场景有替代:`Shutdown` + 重启进程

如果将来要做,得先想清楚:暂停状态存哪(memory 后端怎么办)、是进程级还是集群级、和 `Shutdown` 的关系。

### 9.3 消息过期丢弃(`msg_expire_seconds`)

funboost 支持"这条消息发布超过 N 秒还没被消费就直接丢弃"(`base_consumer.py:551-559`),按 `publish_time` 判断,超期直接 ack 丢掉不执行。

**价值**:LLM 场景里"排队排了两小时的任务,用户早走了"是真实存在的。

**但挂起**,理由:
- 成本确实低(`SubmitOption` 加个 `ExpireAfter(d)`,认领时判一下)
- 但没有诉求方,且**语义要想清楚**:过期的任务进什么状态?`canceled`?新状态 `expired`?进 `canceled` 会和用户主动取消混淆(状态查询分不清);加新状态要动状态机和五后端
- 等真有人要再定

## 10. 明确不做(维持既有裁决)

funboost 有、但 taskgate 明确不做的,**本次调研不构成重新讨论的理由**:

| 项 | 既有裁决出处 |
|---|---|
| 40+ broker 支持 | 五后端过同一套契约是有意的克制;funboost 的 40+ 里相当比例是套壳 |
| Web UI / 管理后台(funweb) | 方案 v5 第 10 节 |
| FaaS / CDC / 爬虫 / MQTT | 全部超出"库"的定位。**注意与路线图的关系(v4 澄清)**:"不做 FaaS"指的是 **taskgate 库本身不变成 FaaS 平台**;另建独立平台并**内嵌 taskgate 库**做可靠投递(如 `2026-07-16-Serverless函数计算平台设计方案.md` 的 Trigger Service)不在此列——那正是"库"定位的正确用法,两份方案不打架 |
| Workflow 声明式编排(Chain / Group / Chord) | 方案 v5 第 13 节已论证不做 DAG,`DependsOn` 的克制版是有意的 |
| 任务优先级调度 | 方案 v5 第 11 节标"(可选)",M3 完成记录复核后维持不做 |
| webhook 通知 | 方案 v5 第 10 节,M3 复核维持不做 |
| cron 周期调度 | 方案 v5 第 10 节,给了更好的替代配方(见 4.3) |
| 远程强杀任务 | funboost 用 ctypes 杀线程,自己承认可能死锁;taskgate 的 ctx cancel 更干净 |
| 独立部署 server 模式 | 就是个库 |

另外,funboost 的**异常分类**(`ExceptionForRetry` / `ExceptionForRequeue` / `ExceptionForPushToDlxqueue`)和 taskgate 的 `ErrThrottled` / `ErrSkipRetry` 是同一思路,**已经对齐,不用动**。

## 11. 反面教材(写给后续设计参考)

funboost 为"功能全"付出的代价,值得记下来:

1. **配置膨胀**:`BoosterParams` 把 40 多个参数塞进一个类,从 `queue_name` 到 `is_fake_booster` 到 `booster_registry_name` 全混在一起,还得靠注释区分"普通用户不用管不用改"。contrib 扩展又只能通过 `user_options` 这个无类型 dict 传配置。**Go 里更不该学**——这也是本方案对周期配额和熔断都强调"走能力接口,别往 `QueueConfig` 堆字段"的原因。
2. **拼写错误固化成 API**:`priroty`、`hearbeat`、`msg_expire_senconds`、`consumin_function_decorator`、`cousumer`、`SQLACHEMY`。为了向后兼容还专门做了一层 rename 映射(`core/func_params_model.py:65`)。**早期命名错误的长期成本。**
3. **单文件 1601 行**:`base_consumer.py` 里 `_run` 和 `_async_run` 大面积重复(作者在 `:1024-1049` 专门解释为什么无法合并——Python 不能在同步函数里条件性 `await`)。Go 没有这个二分问题。

**但有一点值得学**:funboost 把 OTel、Prometheus、熔断、周期配额、微批**全做成可选 mixin**,核心代码里没有任何这些的痕迹,不用的人零负担(包体积和依赖都不受影响)。

**taskgate 已经有这个模式的先例**:`LimiterProvider` 就是可选能力接口,scheduler 只做一次断言、不 import 任何后端包。周期配额应当沿"能力接口"这个**模式**走,但接口本身大概率要新开或拆分(现有 `LimiterProvider` 是整包替换,配额直接复用它不成立,见 5.4);熔断将来若做同理。

### 已知风险

1. **任务身份语义(6.4)**:ID 复用会静默篡改历史依赖指向。五个候选(禁止释放 / 级联删 / generation / 拆两个概念 / 原地重跑 `Retry`/`Replay`)代价差异很大,倾向先评"原地重跑"(不发生 ID 复用,问题从根上消失)再决定要不要通用清理 API,但没把握
2. **`Delete` 与已落地的 MySQL/PG 冲突(6.5d)**:`docs/plans/2026-07-15-MySQL-PG后端适配方案.md` 的"结论先行"第 1 条写着「Broker 接口一行不用改」,而 pg/mysql/`internal/sqlbroker` **已经落地**。加 `Delete` 就是打破它——**这不再是假设,该文档必须回头修订**
3. **配额的产品语义(5.4.2)**:"不白扣、不超发、故障可恢复、后端无关"四选三。选"精确认领配额"会破坏 Broker 接口的后端无关性,这一项性价比要重估
4. **配额在多进程后端的可信度(5.4.3)**:若最终选"只有 redis 提供集群级配额",则**功能名和 README 不得宣称是网关硬配额**——否则又是一次"说了假话"
5. **brokertest 契约 2 不动是刻意的**:选 B(`Delete`)或 e(Retry)都不削弱"同 ID 二次入队 → ErrTaskExists,且原任务原封不动"这个好合同。若 6.4 裁决转向 A/C 要重新评估;**选 f 也要重新评估**——幂等判定从"任务 ID"改到"业务键"(同一业务键最多一个非终态 execution),契约 2 的表述要跟着领域模型重写
6. **第 7 节的措辞改动会让 README 看起来"变弱"**:把"恰好执行一次"改成"at-least-once,handler 必须幂等"读起来像功能缩水。但这是**如实描述**——能力一点没变,变的只是不再说假话。要顶住"这样写不好卖"的直觉。dask 的 `caching.rst:154-165` 敢在官方 feature 文档里写"这可能会 OOM 崩掉你的会话",诚实的文档只会赢得信任
7. **本方案 v1 的教训**:13 条评审全部成立,其中 3 条是源码层面的事实错误。**后续讨论调度行为必须引 `broker.go`/`scheduler.go` 的实现,不能引 README 的描述性文字反推**(#8 就是这么错的)
