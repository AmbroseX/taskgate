# taskgate MySQL / PostgreSQL 后端适配方案

日期:2026-07-15
状态:待人工审查(审查通过后走 spec-kit 流程编码,预计为 spec 004)
前置:M1/M2/M3 已完成,三后端(memory/sqlite/redis)过同一套 brokertest 18 条契约

## 1. 结论先行

> **修订注记(2026-07-16,spec 005)**:下面第 1、2 条在本方案落地(spec 004)时成立,但已被 spec 005(Identity 身份模型)按预定流程扩展:Broker 接口新增第 16 个方法 `Replay`,brokertest 从 18 条扩到 22 条(契约 2 重写为 BusinessKeyIdempotent),`tasks` 表加 `business_key`/`replay_of` 两列与两个唯一约束(PG 部分唯一索引 / MySQL 生成列唯一索引),存量表由 Init 幂等迁移。这不是本方案的错误——"接口不改"描述的是 spec 004 自身的改动面;后续接口演进照常走"brokertest 先行、五后端同批"的纪律。详见 `specs/005-identity-replay/`。

1. **Broker 接口一行不用改**。接口本来就是按"所有后端都能同语义实现"设计的,MySQL/PG 都是事务型数据库,15 个方法(含 Init/Close)全部能实现,语义和 sqlite 完全对齐。
2. **验收标准现成**:新后端过 brokertest 18 条契约就算合格,不需要新契约。
3. **实现蓝本是 sqlitebroker,但"照抄"有两个陷阱**:sqlite 是单连接串行,蓝本里没有的**死锁重试**和**占位符复用限制**是 SQL 后端必须新写的机制,而且直接决定 Dialect 接口签名和 withTx 结构(见 4.1/4.2,research 阶段的头号问题)。其余表结构、事务边界、SQL 骨架 90% 可复用。
4. **推荐包结构**:共享核心 `internal/sqlbroker` + 两个薄壳公开包 `pgbroker/`、`mysqlbroker/`(理由见第 3 节)。
5. **测试没有 miniredis 那样的纯 Go 替身**,契约测试走环境变量门控(和真 Redis 档一个路子),CI 用 service container 提供真库。代价:这是首个"本地 go test 全绿但实际没跑过"的后端,回归只能靠 CI 兜——这一点连同门控例外一起写进宪法修订。
6. **建议 PG 先做,MySQL 跟上**:PG 支持 `UPDATE ... RETURNING`,代码形态和 sqlite 最接近,风险最小。

## 2. 为什么适配成本低:现有设计里已经铺好的路

| 现有设计 | 对 MySQL/PG 的意义 |
|---|---|
| Broker 接口签名冻结、后端无关 | 新后端零接口改动,scheduler/client 一行不动 |
| brokertest 契约套件(工厂函数注入) | 验收现成:`brokertest.Run(t, factory)` 挂上即测 |
| 所有时间由 Go 层 `clock.Now().UnixMilli()` 传入,SQL 里不用 NOW() | fakeclock 直接可用,契约测试确定性不受影响 |
| 时间戳统一毫秒精度(M3 定型,有并列契约压着) | BIGINT 存毫秒,三库行为天然一致 |
| "终态更新+子任务唤醒"同事务(宪法 III) | MySQL/PG 都是真事务库,照抄 sqlite 的事务边界即可 |
| LimiterProvider 是可选能力接口 | SQL 后端先不实现,scheduler 自动退回进程内限流,零改动 |
| Dequeue = 内部唤醒信号 + 100ms 兜底轮询 | 结构照搬,跨进程写入靠轮询发现(和 sqlite 跨进程一样) |

## 3. 包结构:共享核心 + 薄壳(推荐)

### 方案 A(推荐):`internal/sqlbroker` 核心 + 薄壳公开包

```text
taskgate/
├── internal/sqlbroker/     # database/sql 通用实现,注入 Dialect
│   ├── broker.go           # 结构体、Init/Close、withTx、wake、scanRec(从 sqlitebroker 迁移适配)
│   ├── dialect.go          # Dialect 接口(见下)
│   ├── enqueue.go / dequeue.go / lifecycle.go / propagate.go / query.go
├── pgbroker/
│   └── broker.go           # Open(dsn, opts...):blank import pgx 驱动 + PG 方言,~100 行
├── mysqlbroker/
│   └── broker.go           # Open(dsn, opts...):blank import mysql 驱动 + MySQL 方言,~100 行
```

Dialect 接口只收"两库真正不同"的点,保持最小:

```go
type Dialect interface {
    Name() string                       // "postgres" / "mysql"
    Rebind(sql string) string           // ? → $1..$n(PG);MySQL 原样返回
    SchemaSQL(prefix string) string     // 各自的 DDL
    Claim(...) (*rec, error)            // 认领语句(两库形态不同,见 4.3)
    IsDuplicateKey(err error) bool      // PG SQLSTATE 23505 / MySQL errno 1062
    IsRetryable(err error) bool         // 死锁/序列化失败判定,见 4.1(PG 40P01/40001,MySQL 1213/1205)
    Lock(ctx, conn, name) / Unlock(...) // 建表期互斥,见 5 节冷启动竞态(PG advisory lock / MySQL GET_LOCK)
}
```

**共享 SQL 的书写纪律(定死,不然共享模型会被 MySQL 击穿)**:一律只用**位置 `?` 且不复用**——sqlite 蓝本里 `?1`/`?2` 具名复用的写法(如 ReapExpired 里 nowMS 被引用 3 次)迁移时必须展开成多个 `?`,同一个值在 args 里重复传。原因:`Rebind` 只能改 SQL 文本改不了 args 切片,go-sql-driver/mysql 只认位置参数不支持复用,PG 的 `$n` 复用反而是特例。宁可 args 里重复传值,不把 Rebind 升级成"文本+参数"双改写(复杂度不值)。队列过滤统一用现成的 `placeholders(n)` 展开 `IN (?,...)`,PG 不特开 `ANY(array)` 绑定。

**理由**:enqueue/lifecycle/propagate/query 的 SQL 是标准 SQL,两库几乎逐字相同,复制两份就是双倍维护面(每次行为契约变更要同步改两处,漂移风险正是宪法 II 要防的)。差异集中且可枚举,Dialect 收口住即可。薄壳包各自 blank import 自己的驱动——用户只用 PG 就不会把 MySQL 驱动链进二进制。

### 方案 B(备选):两个独立包各自照抄 sqlitebroker

更符合 KISS,但 ~7 个文件 × 2 份的 SQL 复制,后续每条契约变更都要三处同步(sqlite/pg/mysql),长期看违背 DRY 且容易漂移。**不推荐**,除非审查时判断 Dialect 抽象过度。

### 明确不动的

- `sqlitebroker/` 保持原样,**不**往 sqlbroker 里合并:它的并发模型(单连接串行 + BEGIN IMMEDIATE)和服务器型数据库(连接池 + SKIP LOCKED)根本不同,硬塞一个方言里会让 Dialect 接口膨胀。
- memory/redis 后端零改动;scheduler/client/brokertest 零改动(只加测试接线)。

## 4. 相对 sqlite 蓝本要新写/要改的技术点

### 4.1 死锁/序列化失败自动重试(蓝本没有的新机制,research 头号问题)

sqlite 单连接串行,天生不会死锁,所以蓝本里一行死锁处理都没有——这是照抄的最大陷阱。PG/MySQL 是真并发行锁,多行事务必然产生死锁窗口:

- `propagateTx` 在一个事务里按 parent_id 锁边、再逐个锁子任务行,**加锁顺序取决于树的形状**。两个并发操作(Cancel 取消子树 vs ReapExpired 回收租约、两个 Fail 传播进同一片子树)加锁顺序不同 → PG 报 40P01、MySQL 报 errno 1213,事务被数据库单方面 kill。
- 认领的 SKIP LOCKED 只避开"认领 vs 认领",避不开"认领 vs 传播""传播 vs 传播"。

不处理的后果:连锁取消/并发契约在真库上偶发红,是"CI 跑 100 次挂 3 次"级别的脏 flaky。

**方案**:`withTx` 内置重试环——fn 报错时用 `Dialect.IsRetryable` 判定,可重试则回滚后指数退避重跑整个 fn,上限 N 次(建议 5 次,退避走注入 clock);超限或不可重试原样返回。三条配套纪律(全部写进 research):

- **fn 必须无副作用可重跑**:每轮重新 getRecForUpdate,蓝本骨架本来就满足。
- **Notify 严禁在重试环内触发**:fn 只把状态流转**快照收集**进切片,`fireNotify` 只在 withTx 整体成功返回后调用——否则前几轮 rollback 掉的事务已经把通知发出去了,等于对外广播了从未发生的状态流转(幻通知)。sqlite 蓝本已是"事务外异步调"的结构,但没有重试环压力,这里必须写成硬性纪律并在 code review 时逐调用点核对。
- **MySQL 1205 不能和 1213 同等重试**:1213(死锁)是立即返回,重试便宜;1205(lock wait timeout)默认要等 `innodb_lock_wait_timeout=50s` 才报,再指数退避重试 5 次,单次调用最坏挂死几百秒。处理:连接初始化时 `SET SESSION innodb_lock_wait_timeout` 调低(如 5s),且 1205 单独限重试 1 次不退避;IsRetryable 因此可能需要区分"立即重试"和"限次重试"两档,research 定形态。PG 的 40P01/40001 都是立即返回,不受此影响。

### 4.2 占位符纪律(蓝本的 `?1` 复用写法必须展开)

见第 3 节"共享 SQL 的书写纪律":只用不复用的位置 `?`,值重复就在 args 里重复传;迁移 sqlite SQL 时逐条检查 `?N` 复用点(ReapExpired 是重灾区)。这条决定了 Rebind 能保持"只改文本"的最简形态。

### 4.3 认领语句

sqlite 靠"整库单连接串行 + BEGIN IMMEDIATE"避免认领竞争;MySQL/PG 是真并发连接池,必须用 `FOR UPDATE SKIP LOCKED`(要求 MySQL 8.0+ / PG 9.5+)防止两个 worker 认领同一行、也不互相等锁:

PG(有 RETURNING,和 sqlite 形态几乎一样,一条语句完成):

```sql
UPDATE tasks SET status='running', lease_token=$1,
       started_at = CASE WHEN started_at=0 THEN $2 ELSE started_at END
WHERE id = (SELECT id FROM tasks
            WHERE queue IN ($3,...) AND status IN ('pending','retrying') AND run_at <= $n
            ORDER BY run_at, id LIMIT 1
            FOR UPDATE SKIP LOCKED)
RETURNING <taskCols>;
-- lease_until 和 sqlite 一样同事务第二条 UPDATE 补记:TTL 按队列配置,
-- 多队列认领时 SET 那一刻还不知道中选的是哪个队列,一条语句写不进正确的 TTL。
```

MySQL(没有 UPDATE...RETURNING,同一事务里拆两步,SKIP LOCKED 保证第二步不会被抢;SELECT 先拿到 queue,lease_until 倒是可以在 UPDATE 里一步写全):

```sql
-- 同一事务内:
SELECT <taskCols> FROM tasks
WHERE queue IN (?,...) AND status IN ('pending','retrying') AND run_at <= ?
ORDER BY run_at, id LIMIT 1 FOR UPDATE SKIP LOCKED;
UPDATE tasks SET status='running', lease_token=?, lease_until=?, started_at=... WHERE id = ?;
```

### 4.4 连接池与事务

- 不再收紧到 1 个连接,但也不能放任:database/sql 的默认是**不限连接数**,一堆 worker 的阻塞 Dequeue 轮询会打爆服务器 `max_connections`。`SetMaxOpenConns` 给保守默认(10),`Options{MaxOpenConns}` 可调;并发安全由 SKIP LOCKED + 行锁 + 4.1 的死锁重试共同保证。
- 隔离级别:两库默认(PG Read Committed / MySQL Repeatable Read)都够——所有写路径先锁行再判状态改状态,和 sqlitebroker 的"事务内 getRec → checkLease → UPDATE"骨架一致。
- **行锁只进写事务,不泄漏到读路径**:拆两个取行函数——`getRecForUpdate`(SELECT ... FOR UPDATE,只在写事务里用)和普通 `getRec`(Get/List 等只读路径直接走连接池、无事务无锁)。无脑全加 FOR UPDATE 会让只读的 Get 在自动提交连接上白加行锁。
- ctx 取消:database/sql 会中断在跑的语句,错误以驱动私有形态漏出;照抄 sqlite 的处理——出错后先查 `ctx.Err()`,非空就统一返回 ctx 错误(契约要求)。

### 4.5 表结构差异

时间列继续 BIGINT 存 unix 毫秒、由 Go 层传入,这条纪律不变。差异点:

| 列 | sqlite | PG | MySQL |
|---|---|---|---|
| id / type / queue | TEXT | TEXT | VARCHAR(255)(TEXT 不能做主键/索引) |
| payload / result | BLOB | BYTEA | LONGBLOB |
| 各时间/计数 | INTEGER | BIGINT | BIGINT |
| 排序规则 | 默认(二进制) | 默认(C/字节序即可) | **必须 utf8mb4_bin**(ID 大小写敏感,默认 _ci 会把自定义 ID "abc"/"ABC" 判成重复,排序契约也会漂移) |

两个 MySQL 独有的行为差异,必须在入口挡住而不是让驱动报脏错(sqlite/PG 的 TEXT/BYTEA 没有这两个限制,契约 18 条也测不出来):

- **长度上限**:VARCHAR(255) 意味着超 255 字符的自定义 ID/type/queue 写不进。处理:mysqlbroker 的 Enqueue 入口显式校验长度、超限返回清晰错误(参照 redisbroker `validateID` 挡控制字符的先例),并写进已知限制"MySQL 后端 ID/type/queue 最长 255 字符"。
- **payload/result 大小**:LONGBLOB 受 `max_allowed_packet` 限制(默认 64M),超了在协议层报错。不做入口校验(上限是服务器配置,客户端不可知),写进已知限制提醒用户。

- 表名与**索引名/约束名**都带前缀:服务器库常被多应用共享,默认 `taskgate_tasks` / `taskgate_task_deps`,提供 `Options{TablePrefix}`(对齐 redisbroker 的 KeyPrefix 思路)。索引名不能漏——PG 的索引名在 schema 内全局唯一,两个应用裸用 `idx_claim` 会撞名,统一 `<prefix>idx_claim` 这类形态。
- 建表:沿用 sqlite 的 `CREATE TABLE IF NOT EXISTS` 自动建表,不引入迁移框架(YAGNI)。**但要处理冷启动竞态**:多个进程同时首启并发跑 DDL,PG 会报 `tuple concurrently updated`/`duplicate key pg_type` 这类脏错(PG 已知问题,sqlite 单连接没这事)。建表语句外面套一把库级互斥锁——PG 用 `pg_advisory_xact_lock`、MySQL 用 `GET_LOCK`(收进 Dialect 的 Lock/Unlock),并对这类错误再兜一层容忍重试。**锁必须钉在独占连接上**:MySQL 的 GET_LOCK 是会话(连接)级的,走连接池会出现"锁在 A 连接上拿的、RELEASE_LOCK 在 B 连接上放"的错位——用 `db.Conn(ctx)` 取一个独占连接,加锁 + DDL + 解锁全在它上面跑完再归还;PG 的 `pg_advisory_xact_lock` 事务结束自动释放,同样要求锁和 DDL 在同一连接的同一事务里。
- 索引照抄:`(queue, status, run_at)` 认领索引、`(status, lease_until)` 收割索引、task_deps 的 `(parent_id, done)`。

### 4.6 重复键 → ErrTaskExists

Enqueue 撞主键时要翻译成 `taskgate.ErrTaskExists`。判定驱动相关:PG 看 SQLSTATE `23505`,MySQL 看 errno `1062`,收进 Dialect 的 `IsDuplicateKey`。

实现提示(IsDuplicateKey/IsRetryable 共同适用,写进 Dialect 注释):错误判定一律 `errors.As` 到驱动的具体错误类型(`*pgconn.PgError` / `*mysql.MySQLError`)再看 SQLSTATE/errno,禁止字符串匹配——错误文案随驱动版本和 locale 变,字符串匹配就是定时炸弹。

## 5. 不用改但要确认对齐的点

- **唤醒与阻塞 Dequeue**:照搬 sqlite 的三唤醒源结构(同进程 wake channel / 注入 clock 的到点等待 / ctx)。跨进程写入靠 100ms 兜底轮询——对服务器库这是一次网络往返,默认间隔建议放宽到 200ms 并开 Option 可调;PG 的 LISTEN/NOTIFY 能做即时跨进程唤醒,但要独占连接、增加故障面,**先不做**(YAGNI,记入已知限制)。
- **依赖传播/连锁取消**:sqlitebroker/propagate.go 的"同事务内工作队列迭代收敛整棵子树"是纯标准 SQL,两库直接复用;宪法 III.4(v1.1.0)允许的就是这种单事务整树形态。
- **ReapExpired**:同 sqlite 逻辑,注意扫描语句也要 `FOR UPDATE SKIP LOCKED`(多进程各自跑 reaper 时不互踩、不重复计 LeaseLost)。
- **List 分页**:`ORDER BY created_at, id LIMIT ? OFFSET ?` 两库原生支持,比 redis 还省事;毫秒并列契约(第 18 条)天然满足。

## 6. 测试策略(需要宪法小修的唯一点)

MySQL/PG 没有 miniredis 那样能进 CI 的纯 Go 替身(go-mysql-server 不支持 SKIP LOCKED,embedded-postgres 要联网下二进制,都不满足"离线确定")。方案:

- **契约档(L2)**:`TASKGATE_PG_DSN` / `TASKGATE_MYSQL_DSN` 环境变量门控,缺失即 skip——完全复制真 Redis 档(`TASKGATE_REDIS_ADDR`)的既有模式,含随机表前缀隔离 + 测试后清理。
- **CI**:GitHub Actions service container(postgres:16、mysql:8)注入 DSN,契约在 CI 必跑;本地开发者不装库也能全绿(skip)。
- **L3/多进程专项**:`forEachBackend` 系列在 DSN 存在时追加两个后端;kill -9 / 恰好一次 / 断连专项对 SQL 后端同样有效(复用 tcpProxy)。
- **并发 propagate 压测(新增专项,验证死锁重试环)**:4.1 是本方案自认的头号机制,但契约用例基本单 goroutine 顺序跑,压不出死锁——不加专项它就没有任何自动化证明。L3 加一条:构造一片共享子树,多个 goroutine 同时打 Cancel/Fail/ReapExpired 进去,断言零错误抛出、全部任务终态一致(死锁被重试环吃掉而不是漏给调用方)。env 门控,与契约档同一开关。
- **宪法修订(MINOR)**:V 条"测试离线可跑"补例外——"没有纯 Go 内存替身的后端,契约测试允许环境变量门控 + CI service container,本地缺环境时 skip 不算红"。**修订文案必须把代价明写出来**:这是首个本地零覆盖的后端(redis 好歹有 miniredis 兜底),本地 `go test -race` 全绿不代表跑过 PG/MySQL,这两个后端的回归唯一防线是 CI——不能只写"允许 skip"让本地绿的含金量悄悄缩水。走 /speckit-constitution 记修订记录。

## 8. 分布式限流的取舍

MySQL/PG 是跨进程共享介质,理论上能实现 LimiterProvider(信号量槽位表 + 租约 TTL)。**第一期不做**:

- scheduler 对没有 LimiterProvider 的后端自动退回进程内限流,行为和今天的 sqlite 后端完全一致,多进程场景各进程各限各的(写进已知限制);
- 用数据库行锁做高频限流信号量,每次 Acquire/Release 都是一次事务写,吞吐上不划算——真需要跨进程精确限流的用户,方案里本来的答案就是 redis 后端;
- 后续真有需求,能力接口留着口子,不动任何既有代码就能加。

## 9. 里程碑与工作量

按 spec-kit 走一个新 spec(建议 `specs/004-sql-brokers/`),开发顺序:

1. **specify/plan/tasks**:本文档作为输入,契约零新增。research 阶段的头号问题按序:①withTx 死锁重试环的结构与"fn 可重跑"纪律(4.1);②占位符不复用纪律与迁移点清单(4.2);然后才是 Dialect 边界、认领 SQL、建表互斥、测试门控。
2. **实现顺序**:internal/sqlbroker 骨架(从 sqlitebroker 迁移适配)→ PG 方言 + pgbroker 薄壳 → brokertest 接线全绿 → MySQL 方言 + mysqlbroker → 契约全绿 → L3/多进程接线 → CI service container → README/完成记录。
3. **验收**:brokertest 18 条在 PG/MySQL 全绿(env 档);`go test ./... -race` 本地(无 DSN,skip)与 CI(有 DSN,全跑)双绿;覆盖率新后端 ≥80%。

工作量参照:sqlitebroker 全包约 7 个文件,核心迁移 + 两方言 + 测试接线,体量约等于当年 M2 redis 后端的一半(没有 Lua、没有限流器)。

## 10. 已知限制(随实现写进 README)

- 跨进程新任务感知延迟 = 轮询间隔(默认 200ms,可调);PG LISTEN/NOTIFY 即时唤醒留作后续可选项。
- 不提供分布式限流(LimiterProvider 不实现),多进程各自进程内限流;需要精确跨进程限流用 redis 后端。
- 要求 MySQL 8.0+ / PostgreSQL 9.5+(SKIP LOCKED);MySQL 表必须 utf8mb4_bin 排序规则(建表 DDL 已内置)。
- 契约测试需要真库(env 门控),本地无库时 skip——本地全绿不覆盖这两个后端,回归靠 CI。
- 高并发传播冲突时事务会经历死锁重试(自动、有上限),表现为个别调用延迟抬高;重试超限会把数据库的死锁错误原样抛给调用方。
- MySQL 后端:ID/type/queue 最长 255 字符(Enqueue 入口校验,超限报清晰错误);payload/result 受服务器 `max_allowed_packet` 限制(默认 64M),超限由驱动报错。
