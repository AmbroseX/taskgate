## 2. 相对方案的两处实现优化(经审查采纳)

1. **认领与收割统一两步式,取消 dialect.Claim 与 RETURNING 抽象**。方案原设想 PG 用 `UPDATE...RETURNING`、MySQL 两步式,还要为 ReapExpired 的多行 RETURNING + `||`/CONCAT 差异做抽象。实现发现:认领和收割统一用「`SELECT ... FOR UPDATE SKIP LOCKED` 拿行 → 逐行 UPDATE」,这个语法 PG/MySQL 完全一致,失败消息在 Go 里 `fmt.Sprintf`,于是 **Dialect 从方案的 8 方法缩到 7 个**(Name/Rebind/SchemaSQL/IsDuplicateKey/Retryable/Lock/Unlock),不需要 Claim、不需要 RETURNING/CASE/CONCAT 方言分支。更 KISS/DRY。

2. **入队对父行加 `FOR UPDATE` 锁**(方案未明确要求)。见第 3 节的竞态修复。

## 3. 真机验证中发现并修复的竞态(关键)

**现象**:L3 的 `TestFanInDependency`(扇入:C 依赖 A、B,两父完成后 C 应被唤醒)在真 MySQL 上 2/3 概率超时失败;memory/sqlite/redis 全过。

**根因**:C 入队时读父状态用不加锁的 `SELECT`。当「父完成后向子传播减计数」与「子入队登记依赖边」并发交错时,父的传播 `childrenOf` 读不到子尚未提交的依赖边(不给子减计数),而子又带着满额 pending_parents 提交、父不会再传播 → 子永远卡 blocked。sqlite 靠单连接串行(SetMaxOpenConns(1))天然避开,memory/redis 靠原子临界区,MySQL/PG 真并发就暴露了。

**修复**:入队时对每个父行 `SELECT status ... FOR UPDATE`,强制「父完成传播」与「子入队」二选一严格先后,死锁交给 withTx 重试环。修复后扇入用例连跑 5/5 稳定通过,全套 L3 对真 MySQL 全绿。

**教训**:这是只有真库并发才能暴露的竞态,契约用例(单 goroutine)与本地无库都测不出——正印证方案「回归唯一防线是 CI」的判断。

## 4. 死锁重试环验证(方案头号机制,SC-005)

新增 `TestConcurrentPropagate`:构造钻石依赖(一批子任务同时依赖两个父),并发 Fail 两父 + 两个 reaper 搅动,20 轮。真 MySQL 上观测到**死锁重试环每轮触发 46–50 次**,全部被吃掉,零错误泄漏给调用方,所有任务终态一致。证明重试环真实生效(不是空绿)。三态 RetryClass:PG 40P01/40001 与 MySQL 1213 走立即重试,MySQL 1205(锁等待超时)走有限重试 1 次 + 会话级 innodb_lock_wait_timeout=5s。

## 6. 已知限制(已写进 README)

- 跨进程新任务感知延迟 = 轮询间隔(默认 200ms 可调);未做 PG LISTEN/NOTIFY。
- 不提供分布式限流(不实现 LimiterProvider,退回进程内限流);需精确跨进程限流用 redis。
- 高并发传播冲突会经历死锁自动重试(上限 5 次),个别调用延迟抬高;超限原样抛死锁错误。
- MySQL:ID/type/queue ≤255 字符(入口校验);payload/result 受 max_allowed_packet(默认 64M);表 utf8mb4_bin。
- 契约测试需真库(env 门控),本地无库 skip;PG/MySQL 本地零覆盖,回归靠 CI(宪法 V.5)。
- 要求 MySQL 8.0+ / PostgreSQL 9.5+(SKIP LOCKED)。
